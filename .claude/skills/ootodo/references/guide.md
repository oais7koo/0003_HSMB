## 1. 개요

`ootodo`는 d0004_todo.md의 "대기 ToDo" 섹션을 관리하고 업무를 자동 처리하는 스킬입니다.

**핵심 가치**: Todo 추가와 처리를 한 번에

---

## 2. 빠른 시작

### 2.1 가장 많이 쓰는 명령어

```bash
# 추가 + 즉시 처리 (권장)
ootodo "버그 수정"

# 상태 확인
ootodo status
```

### 2.2 서브명령어 요약

| 명령어 | 설명 |
|--------|------|
| `ootodo "내용"` | 추가 + 즉시 처리 (권장) |
| `ootodo` | 대기 중 업무 전체 처리 |
| `ootodo status` | 서브명령어 리스트, 대기 목록 |
| `ootodo add "내용"` | 추가만 (나중에 처리) |
| `ootodo clear` | 완료 항목 → d0010 아카이브 |

## 3. 상세 사용법

### 3.1 추가 + 즉시 처리 (권장)

```bash
ootodo "로그인 API 버그 수정"
```

**워크플로우**:
1. d0004_todo.md "대기 ToDo" 섹션에 항목 추가
2. 작업 유형 판별 (코딩/비코딩)
3. 에이전트가 즉시 처리
4. 완료 시 "완료" 섹션으로 이동

### 3.2 추가만 (나중에 처리)

```bash
ootodo add "새 기능 구현"
ootodo add "긴급 수정" --priority high
ootodo add "문서화" --note "v2.0용"
```

### 3.3 일괄 처리

```bash
# 대기 중 업무 전체 처리
ootodo

# 최대 3개만 처리
ootodo --max-items 3
```

### 3.4 완료 항목 아카이브

```bash
# 완료 항목 → d0010_history.md 이동
ootodo clear

# 미리보기만
ootodo clear --dry-run
```

**권장 주기**: 1~2주마다 실행

## 4. 작업 유형 판별

ootodo는 내용을 분석하여 자동으로 처리 방식을 결정합니다.

### 4.1 판별 룰셋

| 유형 | 키워드 | 처리 방식 |
|------|--------|----------|
| 코딩 | 구현, 개발, 수정, 버그, 리팩토링, 테스트, API, 함수 | `oodev run` |
| 비코딩 | 문서, 설명, 작성, 정리, 분석, 검토, 다이어그램 | 직접 처리 |

### 4.2 비코딩 작업 처리

| 작업 유형 | 처리 방법 |
|----------|----------|
| 문서 작성/수정 | 해당 문서 읽기 → 내용 추가/수정 → 저장 |
| 설명 추가 | 대상 섹션 확인 → 상세 설명 작성 |
| 분석/검토 | 대상 읽기 → 분석 결과 정리 → 문서화 |

## 5. TODO 기록 형식 표준

### 5.1 블록 포맷 (표준)

**대기 ToDo 블록**:
```markdown
### C001 [tag] 항목 제목
#### 등록일: 2026-04-19 | 우선순위: medium
#### ToDo 내용
내용 텍스트 (없으면 다음 헤더 바로 이어짐)
```

**완료 ToDo 블록**:
```markdown
### C001 [tag] 항목 제목 ✅
#### 등록일: 2026-04-19 | 우선순위: medium
#### ToDo 내용
원래 내용 (있을 경우)
#### 완료일: 2026-04-19
#### 완료 내용
완료 처리 내용
```

**포맷 규칙**:
- 헤더(`####`)와 내용 사이 **빈 줄 없음** (내용이 없으면 다음 `####` 바로 이어짐)
- 블록 간 **빈 줄 1개** 유지
- `ootodo add` → 대기 블록 생성, `ootodo complete [ID]` → 완료일 자동 기록 후 완료 섹션으로 이동

### 5.2 형식 구조

| 필드 | 형식 | 설명 |
|------|------|------|
| ID | `### [접두사]` + 3자리 헤딩 | Todo/이슈 식별자 (C001, T001, A001 등) |
| 등록일 | `#### 등록일: YYYY-MM-DD \| 우선순위: high/medium/low` | 등록 날짜 및 긴급도 |
| ToDo 내용 | `#### ToDo 내용` 헤더 + 본문 | 상세 내용 (선택) |
| 완료일 | `#### 완료일: YYYY-MM-DD` | 완료 시 추가 |
| 완료 내용 | `#### 완료 내용` 헤더 + 본문 | 완료 처리 내용 |
| 완료 섹션 | `## 완료 ToDo` | 완료 시 이 섹션으로 블록 이동 |

### 5.3 우선순위 기준

| 우선순위 | 분류 | 처리 기한 |
|---------|------|----------|
| 긴급 | `high` | 즉시 |
| 일반 | `high` | 24시간 내 |
| 주의 | `medium` | 1주 내 |
| 참고 | `low` | 여유 시 |

## 7. 옵션 상세

| 옵션 | 설명 | 기본값 |
|------|------|--------|
| `--sp N` | 서브프로젝트 지정 | 00 |

| `--max-items N` | 최대 처리 항목 수 | 전체 |

## 8. 워크플로우 다이어그램

### 8.1 추가+즉시처리 플로우 (`ootodo "내용"`)

```
ootodo "내용"
    ↓
d0004_todo.md "대기 ToDo" 섹션에 추가
    ↓
작업 유형 판별
    ↓
┌─────────────┬─────────────┐
│ 코딩 작업   │ 비코딩 작업  │
│ oodev run │ 직접 처리   │
└─────────────┴─────────────┘
    ↓
완료 → "완료" 섹션 이동
```

### 8.2 일괄 처리 플로우 (`ootodo`)

```
ootodo 실행
    -> d{SP}0004_todo.md "대기 ToDo" 섹션 읽기
    -> 대기 중 항목 없음? -> "처리할 업무가 없습니다" 출력 후 종료
    -> 대기 중 항목 있음:
        각 항목에 대해:
        1. 항목을 "진행 중"으로 이동
        2. 스킬 라우팅 룰셋에 따라 해당 스킬 호출 (자동 처리)
        3. 완료 시 "완료" 섹션으로 이동
        4. oocommit run 호출 (커밋 + 이력 이동)
        5. 다음 항목으로 이동
    -> 모든 항목 처리 완료까지 반복
```

### 8.3 상태 전이

```
[대기 중] -> [진행 중] -> [완료] -> [아카이브 (d0010)]
               (실패 시) -> [디버깅 섹션에 등록]
```

### 8.4 check 워크플로우 (`ootodo check`)

```
ootodo check
    1. d{SP}0004_todo.md 로드 (대기 ToDo 전체)
    2. Explore(haiku) — 관련 코드 현재 상태 스캔 (병렬)
    3. ooqa(sonnet) — 이슈 품질·중복·우선순위 분석 (병렬)
    4. python-code-reviewer(sonnet) — 코드 이슈 유효성 검증 (병렬)
    5. 이슈별 검토 의견 출력
    6. 권장 액션 제시
```

**check 출력 형식 예시**:

```markdown
## ootodo check 결과 — SP{N} (2026-04-06)

| ID | 내용 요약 | 현재 우선순위 | 검토 의견 | 권장 액션 |
|----|---------|:----------:|---------|---------|
| T001 | datasets.py getitem 예외 | ERROR | 코드에서 이미 수정됨 (2026-03-18) | 삭제 |
| T002 | api.py 플랫폼 모킹 | WARNING | 유효. 우선순위 낮춤 권장 | INFO로 변경 |
| C001 | 논문 분석 품질 개선 | medium | 유사 항목 C003과 통합 가능 | C003과 병합 |
| C005 | Flutter 넘버링 | medium | 유효. 현재 상태 확인 필요 | 유지 |

소계: 삭제 권장 1건 | 우선순위 조정 1건 | 병합 1건 | 유지 2건
```

### 8.5 update 워크플로우 (`ootodo update`)

```
ootodo update
    1. d{SP}0004_todo.md 로드 (전체 섹션)
    2. 완료 항목 상태 갱신
       - `## 완료 ToDo` 섹션의 블록 확인 → 완료일 미입력 시 오늘 날짜로 채움
       - 디버깅 섹션에서 이미 해결된 이슈 탐지 → 해결일/방법 미입력 시 현재 날짜로 채움
    3. 문서이력관리 버전 업데이트
       - 현재 최고 버전에서 patch(+0.1) 자동 증가
       - 날짜: 오늘 날짜 (YYYY-MM-DD)
       - 변경내용: 이번 세션 주요 완료 항목 요약 (3건 이내)
    4. 미완료 항목 우선순위 재정렬 (기본 활성화)
       - [CRITICAL] > [HIGH] > [MED] > [LOW] 순서
       - 같은 우선순위 내 등록일 오래된 순
    5. 변경사항 저장 + 요약 출력
```

**update 출력 형식 예시**:

```markdown
[ootodo update] 현행화 완료 — SP00

완료 상태 갱신: T032(상태→완료) 1건
이력 추가: v10.3 | 2026-04-17 | T032 완료: Table 12 수치 교정
우선순위 정렬: 완료 (대기 항목 5건 재정렬)

→ d0004_todo.md 저장 완료
```

### 8.6 list 출력 형식 (`ootodo list`)

```markdown
# ootodo list - 전체 대기 업무

| SP | ID | 우선순위 | 내용 (50자 요약) |
|:--:|-----|:-------:|-----------------|
| 00 | T001 | high | ... |
| 01 | I005 | Must | ... |
| 04 | T038 | 높음 | ... |
| 05 | T001 | medium | ... |

총 N건 대기 중 (SP00: X건, SP01: X건, ...)
```

## 9. 트러블슈팅

### Q1: "대기 ToDo" 섹션이 없다고 나옴

**해결**: 섹션이 자동 생성됩니다. d0004_todo.md 파일이 있는지 확인하세요.

### Q2: 처리가 안 됨

**확인 사항**:
1. `ootodo status`로 대기 항목 확인
2. d0004_todo.md 파일 경로 확인
3. `--sp` 옵션으로 서브프로젝트 지정 필요 여부

### Q3: 비코딩 작업이 코딩으로 처리됨

**해결**: 작업 내용에 "문서", "설명", "정리" 등 비코딩 키워드 포함
