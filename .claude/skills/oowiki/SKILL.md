# oowiki - LLM Wiki 지식 체계

> **철학**: "지식을 한 번 컴파일하고 매번 재생성하지 않는다" — Andrej Karpathy
> RAG의 대안: 새 정보 수집 시 LLM이 기존 위키에 **통합**하여 관계를 유지·관리.

## 1. 서브명령어

| 명령어 | 설명 | 출력 |
|--------|------|------|
| `oowiki help` | 서브명령어 목록 표시 | 터미널 |
| `oowiki version` | 스킬 버전 정보 (v10) | 터미널 |
| `oowiki status` | 위키 현황 (페이지 수, 카테고리, 마지막 수집일) | 터미널 |
| `oowiki check` | references/checklist.md 기반 체크 | 터미널 |
| `oowiki run [내용]` | 새 지식 수집·통합 (Ingest) + inbox 감지 + qmd embed | 위키 페이지 + index.md + log.md |
| `oowiki run --url [URL]` | URL 내용 수집·통합 | 위키 페이지 + index.md + log.md |
| `oowiki run --file [경로]` | 파일 내용 수집·통합 | 위키 페이지 + index.md + log.md |
| `oowiki inbox` | 0019_정리 처리 대기 파일 목록 표시 | 터미널 |
| `oowiki inbox --clear` | 0019_정리 전체 파일 삭제 (처리 완료 후) | 터미널 |
| `oowiki inbox --clear [파일명...]` | 0019_정리 특정 파일만 삭제 | 터미널 |
| `oowiki search [질문]` | 위키에서 정보 조회 (RAG 없이 위키 직접 활용) | 터미널 (인용 포함) |
| `oowiki lint` | 모순·고아 페이지·교차참조 누락 점검 | 터미널 |
| `oowiki index` | index.md 전체 재생성 | `0020_wiki/index.md` |
| `oowiki show [페이지]` | 특정 위키 페이지 내용 표시 | 터미널 |
| `oowiki list` | 전체 위키 페이지 목록 | 터미널 |

실행: `uv run python .claude/skills/oowiki/scripts/oowiki_run.py [subcommand] [args]`

## 2. 위키 파일 구조

```
01_obsidian/0020_wiki/
├── index.md          # 전체 카탈로그 (링크, 요약, 메타데이터) — 자동 관리
├── log.md            # 수집·수정·조회 이력 (시간순) — 자동 관리
├── schema.md         # 위키 구조·카테고리 정의
└── [카테고리]/
    └── [페이지명].md # 위키 페이지 (Obsidian 호환)
```

### 위키 경로 설정 (wiki_config.json)

위키 폴더는 컴퓨터마다 경로가 달라, **컴퓨터명(hostname) 기준**으로 해석한다. 설정 파일은 `references/wiki_config.json`:

```json
{
  "wiki_dir": {
    "koodesk": "D:\\resilio\\1_oais\\01_obsidian\\0020_wiki"
  }
}
```

| 우선순위 | 경로 |
|:--------:|------|
| 1 | `wiki_config.json`의 `wiki_dir`에서 현재 컴퓨터명으로 조회 |
| 2 | 기본값 `{프로젝트 루트}/01_obsidian/0020_wiki` |

- **새 컴퓨터 추가**: `wiki_dir`에 `"컴퓨터명(소문자)": "위키 절대경로"` 한 줄 추가. 컴퓨터명은 `python -c "import socket; print(socket.gethostname())"`로 확인.
- **다른 프로젝트에서 중앙 위키 사용**: 위키 콘텐츠는 1_oais 전용 자산이라 다른 프로젝트엔 `01_obsidian/0020_wiki/`가 없다. 그 프로젝트의 `wiki_config.json`에 동일 컴퓨터명으로 1_oais 위키 경로를 지정하면 `oowiki search` 등을 동일하게 쓸 수 있다. inbox(`0019_정리`)도 위키 폴더의 형제 경로로 함께 따라간다.
- **동기화**: `wiki_config.json`은 컴퓨터별 절대경로를 담으므로 oosync로 전파하지 않는다 — Resilio Sync로 컴퓨터 간 동기화된다.

## 3. 핵심 워크플로우

### 3.1 수집·통합 (oowiki run) — Ingest

```
1. [index.md 로드] 기존 위키 카탈로그 파악
   ↓
2. [관련 페이지 탐색] 입력 내용과 관련된 기존 페이지 식별
   ↓
3. [통합 판단]
   ├── 관련 페이지 있음 → 기존 페이지에 새 내용 통합 (단순 추가 X, 유기적 통합)
   └── 관련 페이지 없음 → 새 카테고리/페이지 생성
   ↓
4. [교차참조 갱신] 관련 페이지들 간 [[링크]] 추가/수정
   ↓
5. [index.md 갱신] 새 페이지 또는 수정 페이지 카탈로그 반영
   ↓
6. [log.md 기록] 작업 내용 이력 추가
```

> **핵심 원칙**: 새 정보는 기존 지식에 **통합**된다. 단순 파일 생성이 아닌, 기존 위키와의 관계를 형성하고 교차참조를 유지한다.

### 3.2 조회 (oowiki search)

```
1. index.md에서 관련 카테고리/페이지 식별
2. 관련 페이지 내용 로드
3. 위키 페이지 기반으로 답변 (인용 포함)
4. log.md에 조회 기록
```

### 3.4 inbox 처리 (oowiki inbox)

```
1. [oowiki run] 실행 시 0019_정리 파일 자동 감지·목록 출력
   ↓
2. Claude가 각 파일 내용을 읽어 0020_wiki에 통합 (섹션 3.1 프로세스)
   ↓
3. 통합 완료 후 oowiki inbox --clear 실행 → 0019_정리 파일 삭제
```

> **삭제 안전 원칙**: 삭제는 Claude가 통합을 확인한 후에만 실행. 스크립트는 목록 표시와 삭제만 담당하며 통합 판단은 AI가 한다.

| 명령 | 동작 |
|------|------|
| `oowiki inbox` | 처리 대기 파일 목록만 확인 (삭제 없음) |
| `oowiki inbox --clear` | 0019_정리 전체 파일 삭제 |
| `oowiki inbox --clear 파일명` | 특정 파일만 삭제 |

### 3.5 통합 전 품질 판단 (AI 필터)

> **기본 원칙: 최대 보존 (Maximum Preservation)**
> 사용자가 직접 정리한 내용은 복원이 어려우므로 **누락 없이** 위키에 통합하는 것을 기본값으로 한다. 단, 검색 품질 저하를 막기 위해 **위치·형식**을 신중히 선택한다.

#### 스킵 조건 (정말 통합하지 않을 경우 — 매우 제한적)

| 조건 | 판단 기준 | 처리 |
|------|-----------|------|
| **내용 없음** | 헤더·빈 줄만 존재, 실질적 설명 없음 | 스킵 + log에 기록 |
| **완전 중복** | 기존 위키와 **자구까지 동일** (사례 변형도 없음) | 스킵 + log에 기록 |
| **임시 메모** | "나중에 정리", "TODO", "확인 필요" 등 미완성 메모 | 스킵 + log에 기록 |

> 기존의 "단순 목록", "포맷 전용", "80% 중복" 조건은 제거. **사례 변형이 있으면 통합**한다.

#### 통합 위치 결정 (모두 통합하되 위치를 결정)

```
[질문 1] 이 내용이 기존 위키에 없는 새 개념인가?
  → Yes: 새 페이지 또는 주요 섹션으로 추가
  → No:  아래 질문 2로

[질문 2] 기존 위키와 비교했을 때 어떤 유형인가?
  → 개념 보완·정의 보강:   기존 본문에 유기적 통합 (충돌 시 최신 우선, 기존은 blockquote 보존)
  → 실전 사례·코드 패턴:   "실전 사례" 또는 "추가 예시" 서브섹션에 통합
  → 옵션·세부 표 확장:     기존 표에 행 추가 또는 별도 표 신설
  → 내러티브 설명:         "참고 자료" 섹션에 인용 형식으로 보존

[질문 3] 페이지가 너무 비대해지면?
  → 1,500줄 초과 시 사례 페이지 분리 검토 (예: linux_commands_examples.md)
  → 분리 시 원본 페이지 끝에 [[examples_page]] 링크 유지
```

#### "실전 사례" 서브섹션 형식

각 명령어/개념 페이지의 본문 아래에 다음 형식으로 사례를 누적한다:

```markdown
## 실전 사례

### {간결한 사례 제목}
출처: {원본 PDF/노트}

```sh
# 코드 또는 명령
```

설명: 핵심 의도 1~2줄.
```

#### 통합 시 log.md 기록 형식

```markdown
## YYYY-MM-DD HH:MM — ingest
- **대상**: 파일명 또는 내용 요약
- **결과**: {페이지명} {위치} (개념/사례/표 확장)
- **통합 내용**: 핵심 변경 1~3줄
```

#### 스킵 시 log.md 기록 형식 (드물게 사용)

```markdown
## YYYY-MM-DD HH:MM — ingest (skip)
- **대상**: 파일명 또는 내용 요약
- **스킵 이유**: [조건명] 구체적 이유
```

> **원칙 (수정)**: 정보 보존 우선. 검색 품질은 **위치·형식**으로 관리한다 — 핵심은 위, 사례는 아래 서브섹션. 스킵은 정말 의미 없는 메모일 때만.

### 3.6 개인 프로젝트 코드 지식화 (Code-to-Wiki)

> 코드 프로젝트(0019_정리 등)에서 **개념·패턴·알고리즘** 지식을 추출해 위키화하는 프로세스.
> 코드 파일 자체를 저장하는 것이 아니라, 코드에 담긴 **지식**을 추출한다.

#### 추출 대상 (우선순위 순)

| 우선순위 | 추출 항목 | 추출 방법 |
|----------|-----------|-----------|
| 1 | **프로젝트 개요** | README, 헤더 주석, 파일명에서 목적 파악 |
| 2 | **핵심 알고리즘** | 알고리즘 흐름, 수식, 복잡도 |
| 3 | **중요 특징·제약** | 주석의 "중요" "주의" "TODO", 실패 원인 |
| 4 | **API·라이브러리 사용 패턴** | 핵심 함수 호출 방식, 파라미터 |
| 5 | **입출력 데이터 구조** | 입력 형식, 출력 형식, 전처리 과정 |
| 6 | **성능·실측 수치** | 정확도, 처리 시간, 조건 (보존 필수) |

#### 스킵 조건 (코드 프로젝트 특화)

| 조건 | 판단 기준 |
|------|-----------|
| 설정 파일만 있음 | `.iml`, `.xml`, `.gitignore`, `Makefile` 등만 있는 폴더 |
| 동일 기술 기존 위키 있음 | `bootstrap`, `rnn_lstm` 등 이미 상세 페이지 존재 |
| 데이터 파일만 있음 | `.csv`, `.wav`, `.jpg` 등 데이터·미디어만 있는 폴더 |
| SDK 예제 코드만 있음 | 공식 SDK 예제를 그대로 복사한 경우 (공식 문서가 더 나음) |

#### 위키 페이지 구성 원칙

```
## 개요
[한 줄 목적 + 사용 기술 스택]

## 알고리즘 / 처리 흐름
[단계별 흐름도 or 번호 목록]

## 핵심 코드 패턴
[최소한의 예제 코드 — 개념 이해에 필요한 수준만]

## 실사용 패턴 (oais)
[실제 데이터 형식, 파라미터 선택 이유, 주의사항]

## 성능·수치 (원본 보존)
[실측값은 테이블로 구조화]
```

#### 파일 탐색 순서

```
1. README.md / readme.txt → 프로젝트 목적 파악
2. 헤더 주석 (# 작성자, # 목표, # desc) → 의도 파악
3. 메인 실행 파일 (ps2010_base.py, main.R 등) → 핵심 로직
4. 튜토리얼/예제 파일 → 사용 패턴
5. 데이터 파일 → 입력 형식 파악 (내용 읽기 X)
```

### 3.7 프로젝트 아카이브 처리 (Project-Archive)

> 행정·계약·미팅·보고서 등 **특정 프로젝트의 운영 산출물**(0019_정리에 흔히 쌓이는)을 위키화하는 프로세스.
> 코드 지식화(§3.6)와 달리 **일반화 가능한 지식이 적고 PII·기밀 위험이 높음**.

#### 출력 위치 (위키 외부)

DDC 위키(`0020_wiki/`)에는 넣지 않는다. 분류 시맨틱이 안 맞고 단일 프로젝트 메타가 위키 검색을 오염시킨다. 별도 경로 사용:

```
01_obsidian/0030_projects/
├── README.md                                # 프로젝트 인덱스
├── {YYYY}_{slug}/
│   ├── 01_overview.md                       # 개요·발주처·기간·팀·결과
│   ├── 02_timeline.md                       # 시계열 진행
│   ├── 03_deliverables.md                   # 산출물 요약
│   ├── 04_meetings.md                       # 회의록 요약
│   ├── 05_references.md                     # 참고자료 요약
│   └── 99_admin.md                          # 행정 메타 (PII 없이 사실만)
└── {YYYY}_{slug}_canceled/                  # 중단 프로젝트는 _canceled 접미사
    └── 01_overview.md                        # 중단 사유 명시
```

#### 파일 분류 (4분류)

| 분류 | 정의 | 처리 |
|------|------|------|
| **본 산출물** | 해당 프로젝트가 생산한 보고서·계약서·발표자료·회의록 | 본문 파싱 + 요약 |
| **참고자료** | 본 프로젝트 폴더에 들어있지만 다른 프로젝트의 자료 | 메타·핵심만 — `05_references.md`에 분리 |
| **PII** | 주민증·통장사본·등본·개인 계약서 | 본문 파싱 금지 — 메타(누가/언제/어떤 종류)만 `99_admin.md` |
| **데이터** | CSV·이미지·오디오 등 원본 데이터 | 본문 파싱 안 함. 파일 존재만 기록 |

#### PII 처리 정책 (필수)

| 자료 종류 | 위키 기록 |
|-----------|-----------|
| 주민등록증·등본·통장사본 | "지급증빙 서류 N건 보관 중" — **이름/번호/은행 기록 금지** |
| 프리랜서 계약서 | "프리랜서 계약: {이름}, {계약일}" — 단가·계좌 기록 금지 |
| 법인계약서 | 본문 파싱 OK — 단, 직인·서명·주소 기록 금지 |
| 세금계산서 | "발행: {일자}, 거래내용: {프로젝트명}" — 금액은 사용자 명시 동의 시만 |

#### 파일 크기별 처리 방침

| 크기 | 처리 |
|------|------|
| ≤ 30MB | 본문 파싱 (oohwp/oopdf/python-docx/python-pptx) |
| 30~80MB | 본문 파싱 시도 + 핵심 섹션(개요/결론/목차)만 추출 |
| 80MB+ | **메타만 기록** (제목·작성일·페이지수·추정 내용). 시간/메모리 비효율 |

#### 음성 파일 (M4A/WAV) 처리

| 조건 | 동작 |
|------|------|
| 같은 폴더에 STT 결과(.txt) **존재** | TXT만 사용. M4A 파싱 스킵 |
| STT 결과 **없음** + 사용자가 "STT 처리" 명시 | `ooscrap` 또는 별도 STT 도구로 처리 |
| STT 결과 없음 + 명시 없음 | **메타만 기록** (제목·녹음일자·길이) |

#### 파일 탐색 순서

```
1. 폴더 트리 + 파일명 + 크기 파악 (`find -printf '%s\t%p'`)
2. 4분류 (본 산출물/참고자료/PII/데이터) 자동 분류 + 사용자 검토
3. 본 산출물 작은 것부터 파싱 → 02_timeline.md, 03_deliverables.md, 04_meetings.md 작성
4. 참고자료는 메타 + 핵심만 → 05_references.md
5. PII·행정 → 99_admin.md (PII 정책 준수)
6. 01_overview.md는 마지막에 작성 (3~5번 결과 종합)
```

#### 중단 프로젝트 추가 규칙

| 항목 | 처리 |
|------|------|
| 폴더명 | `{YYYY}_{slug}_canceled` 접미사 |
| `01_overview.md` | "**상태: 중단**" 라벨 + 중단 시점 + **중단 사유** (사용자 확인) 명시 |
| 회의 자료 | 정상 프로젝트와 동일 처리. 중단 직전 회의는 사유 추정 단서 |

> **원칙**: 프로젝트 아카이브는 위키가 아니라 **운영 기록**이다. 검색 가능성보다 정확성·기밀 보호 우선.

## 4. 통합 품질 가이드 (정보 손실 방지)

> `oowiki run` 통합 시 LLM 재구성으로 발생할 수 있는 3가지 위험 패턴과 방어 전략.

### 4.1 다른 개념이 합쳐지는 것 방지

- **한 페이지 = 한 개념** 원칙 엄수 (schema.md 통합 원칙 4번)
- `oowiki run` 시 모호하면 명시적 지시: "새 페이지로 생성" vs "기존 페이지에 통합"
- 비슷해 보이는 개념도 태그/카테고리로 명확히 구분

### 4.2 세부 수치·조건 소실 방지

중요 데이터는 **테이블로 구조화**하면 LLM이 행 단위로 보존한다:

```markdown
## 성능 수치 (원본 보존)
| 출처 | 수치 | 조건 |
|------|------|------|
| 공식 GitHub | 60~90% | 일반 CLI 명령 |
| 실측 (30분 세션) | 80% | 118K→23.9K 토큰 |
```

### 4.3 표현 변형(Paraphrase) 방지

공식 정의·중요 문구는 **blockquote로 고정**하면 LLM이 그대로 유지한다:

```markdown
> "원본 인용문 그대로"
> — 출처
```

### 4.4 공통 안전장치

`oowiki run` 실행 후 **반드시 `git diff`로 변경사항 검토**. 위 3가지를 지켜도 LLM 재구성은 완벽히 막을 수 없으므로 diff 검토가 최후 방어선이다.
