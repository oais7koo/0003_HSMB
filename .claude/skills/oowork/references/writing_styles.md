## ★ 최우선 문장 작성 원칙 (MANDATORY — 모든 스타일 공통)

> 아래 3원칙은 스타일(S01~S04)과 무관하게 **제안서 본문의 모든 문장**에 적용한다.

### 원칙 1. 완전한 문장 구조 유지

주어+서술어 구조를 갖춘 완전한 문장으로 작성한다. 단어·수치 나열 전보체는 금지.

**금지 (전보체):**
```
BM3D 실시간 처리 불가 — 256×256 이미지 0.7초 소요, 차량 탑재 적용 원천 불가 [A003]
```

**허용 (완전한 문장):**
```
BM3D 등 전통 알고리즘은 256×256 이미지 처리에 0.7초가 소요되어 차량 탑재 실시간 적용이 원천 불가 [A003]
```

### 원칙 2. `→` · `—` 기호 없이 한국어 문장 흐름 유지

`→`, `—` 기호로 단어를 연결하는 방식은 금지. 한국어 조사·접속어로 자연스럽게 연결한다.

**금지 패턴 1 — 단어·사실 연결**:
```
DnCNN(60ms·+0.6dB) [A001] → Jetson Orin NX(25W) 특화, 파이프라인 최초 구현
결빙 5년 사고 4,112건·사망 83명 [C011] — 고정 인프라 한정 [C008]
```

**금지 패턴 2 — 라벨 — 설명 (header-detail 형태)**:
> "라벨 — 설명" 패턴도 전보체에 해당하므로 금지. 라벨이 필요하면 한국어 흐름으로 통합한다.
```
PSNR 35→40dB — 4센서 통합 노이즈 제거 정확도 향상 목표
원시데이터 전량 전송 — 1대·1일 1 TB, 1,000대 운영 시 연간 1.34조 원
합산 결과 — 100ms 이내(10 FPS+) 달성 가능
```

**허용 패턴 — 한국어 조사·접속어 흐름**:
```
DnCNN(60ms/512px GPU, BM3D 대비 0.6dB 향상) [A001]을 Jetson Orin NX(25W) 환경에 특화하여
4센서 통합 실시간 노이즈 제거 파이프라인을 최초 구현

4센서 통합 노이즈 제거 단계에서 PSNR을 35dB에서 40dB로 향상시키는 5년차 정확도 목표 설정

원시데이터 전량 전송 시 1대당 1일 1TB가 발생하여 1,000대 운영 시
연간 1.34조 원의 통신·스토리지 비용 추정
```

**예외 (허용)**:
- **표 헤더·셀 안 수치 변화 표기**: `PSNR 35→40dB`, `ODS 0.78→0.85` (도메인 표준 표기)
- **괄호 안 부연 설명**: `Restormer(Zamir et al. CVPR 2022) SIDD >40dB 달성`

> **참고 기준**: `(참고 예시 1) [버스 기반] 핵심기술 개발내용(1.1~1.7.).pdf` 실제 채택 문서 스타일

### 원칙 3. `-` 항목 자기충족성 유지

단순 키워드·명사구 나열은 금지. 각 `-` 항목은 **이유(왜 문제인가)·결과·수치·근거 중 최소 1가지**를 포함하여 독자가 맥락을 파악할 수 있도록 작성한다.

**금지 (키워드 나열):**
```
- 진동·조도·기상 복합 노이즈 미처리
- 단일 RGB 기반 ROI 환경 취약성
- 일률 압축 적용으로 통신·저장 비용 폭증
```

**허용 (이유·결과·수치 포함):**
```
- 진동·조도·기상 복합 노이즈를 미처리한 데이터가 AI 학습에 유입되어 탐지 정확도 저하·오탐 증가 반복 (4센서 통합 SOTA 국내외 부재 [F012])
- 단일 RGB 기반 ROI 추출은 야간·터널·악천후 환경에서 취약하여 이벤트 선별 불가
- 일률 압축 적용으로 통신·저장 비용 폭증 (1,000대 연간 약 1.34조 원 추정 [G002])
```

### 원칙 3-1. 인용 표기 — 3-문서 체계 (70/71/72)

> 정부 R&D 제안서 산출물은 **70(제출용)·71(풀제안)·72(참고정보)** 3개 문서로 분리. 인용 표기 방식은 문서별로 다르다.

| 문서 | 본문 인용 표기 방식 | 예시 |
|------|------------------|------|
| **70_HWP용.md** (제출용) | **자연어 표기만** — 인용 ID 부착 금지 | "Restormer(Zamir et al. CVPR 2022) SIDD >40dB 달성" |
| **71_풀제안내용.md** (풀제안 보존본) | **자연어 표기 + 인용 ID** 부착 | "Restormer(Zamir et al. CVPR 2022) SIDD >40dB 달성 [A002]" |
| **72_참고정보.md** | (자체) — 1부 일반 참고문헌 + 2부 인용 ID별 본문 발췌 | "[A002] Restormer · Zamir et al. (2022) · CVPR · 폴더ID 251106-1027" + 발췌문 |

**핵심 규약**:
- **70 ↔ 71 동기화**: 제안 내용·수치·결론 동일. 차이는 "[A001]" 표기 유무뿐.
- **70은 인용 ID 표기 금지** (제출본 가독성·평가자 시선 우선)
- **71의 모든 [XNNN]은 72에 등재** (G23 매칭 의무)
- **72의 모든 등재 항목은 71에서 1회 이상 인용** (고아 참고문헌 금지)

### 원칙 4. 출처 없는 수치·사실은 인용 금지

본문에 기재하는 **모든 수치·통계·사실 주장**은 반드시 확인된 출처(`[XNNN]` 형식)를 동반해야 한다. 출처가 확인되지 않은 내용은 작성하지 않는다.

> **기술분석(기술연구) 강화 원칙**: 기술·성능·비교 수치 등 모든 기술적 사실은 **논문 또는 URL 원출처**가 실제 존재·확인된 경우에만 작성한다. 원출처 미확보 시 해당 내용을 **언급 자체 금지** — 추정, 요약, 재인용 불가.

**금지 (출처 없는 수치):**
```
○ 도로결빙 사고는 연간 수천 건 발생하며 사망자도 상당수에 달한다
- 엣지 AI 시장은 수십조 원 규모로 빠르게 성장 중이다
```

**허용 (출처 명시):**
```
○ 도로결빙 사고는 5년간 4,112건·사망 83명·치사율 +55% 누적 [C011]
- 글로벌 엣지 AI 시장은 2030년 $66.47B, CAGR 21.7%로 전망 [E001]
```

> **적용 범위**: 수치·통계·타 기관 사례·비교 수치 등 검증 가능한 모든 사실 주장. 자체 산정 수치는 `[GNNN]` + "자체 산정" 표기 필수.

## 등록된 스타일

### S01. 개조식 3단 (기본)

- **용도**: 기술사업계획서, 정부과제 제안서, 공공사업 제안
- **구조**:
  - 1단(□): 소제목·항목 제목 — 핵심 주제/기술명 (KoPub돋움체 Bold 13pt)
  - 2단(○): 중항목 — 세부 내용·목적·범위 기술 (KoPub바탕체 Light 12pt)
  - 3단(-): 소항목 — ○ 항목의 구체적 내용·수치·방법 (KoPub바탕체 Light 11.5pt)
- **어미 규칙 (MANDATORY)**:
  - 명사형 종결만 허용: `~확보`, `~정립`, `~적용`, `~개발`, `~파악`, `~구축`, `~불가`, `~부재`, `~수행`
  - 서술식 종결 금지: `~한다`, `~된다`, `~있다`, `~합니다`, `~이다`
- **배치 규칙**:
  - ○ 뒤 첫 번째 `-` 항목은 **같은 줄 또는 다음 줄** 모두 허용
  - `-` 항목이 여러 개일 경우 이후 항목은 다음 줄에 배치
  - □ 뒤에 ○ 없이 바로 내용이 오는 경우: 해당 내용도 명사형 종결 유지
- **원칙**: 접속사·수식어 최소화, 한 항목에 하나의 핵심 메시지, 정량 수치는 **괄호로 연결** (대시 `—` 사용 금지 — 최우선 원칙 2 적용)
- **실제 예시** (정부과제 제안서 기준):
  ```
  ○ 대표 도로시설물에 대한 기본 제원 정보 (데이터 분류 방안) - 도로시설물 확보 데이터 현황 파악 (정형 및 비정형 데이터 분류)
  - 대상 시설물 확보 데이터 우선순위 선정을 통한 데이터 분류체계 정립
  ○ 디지털 트윈 구축을 위한 2D 도면 확보 및 노후도면 대응 역설계 기술 적용 (드론 및 레이저 스캐닝) - 도로시설물의 도면 부재의 경우 드론 및 레이저 스캐닝을 통한 역설계 기술 적용하여 시설물 제원 정보 확보
  - 데이터 기반 3D 형태 모델 객체 생성을 위한 도로시설물 2D 도면 분석 및 최소단위 객체 유형 확보
  ○ 디지털 트윈 구축을 위한 도로시설물의 이력 데이터 연동 - 도로시설물 보수·보강 및 변경에 대한 이력 데이터 확보 및 객체형 3D 모델에 데이터 탑재를 통한 디지털 트윈 시스템 적용
  ○ 도로시설물 모델링 단순화 - 도로시설물 3D 객체형 모델을 시뮬레이션에 적용하기 위하여 시설물 부재 및 모델 단순화 기능 개발
  ```
- **잘못된 예시 1 (서술식 — 사용 금지)**:
  ```
  □ 실시간 처리 체계 부재: 원시데이터를 전량 전송하는 현행 방식은 1,000대 운영 시 연간 약 1.34조 원의 비용 부담이 추정된다.
  ```
- **잘못된 예시 2 (라벨 — 설명 전보체, 최우선 원칙 2 위반)**:
  ```
  □ 실시간 처리 체계 부재
  ○ 원시데이터 전량 전송 — 1대·1일 1 TB [G001], 1,000대 운영 시 연간 1.34조 원 통신·스토리지 비용 [G002]
  - BM3D 실시간 처리 불가 — 256×256 이미지 0.7초 소요, 차량 탑재 적용 원천 불가 [A003]
  ```
- **올바른 예시 (개조식 + 한국어 흐름 통합)**:
  ```
  □ 실시간 처리 체계 부재
  ○ 원시데이터 전량 전송 시 1대당 1일 1 TB(G001)가 발생하여 1,000대 운영 기준 연간 1.34조 원의 통신·스토리지 비용 추정(G002)
  - BM3D 등 전통 알고리즘은 256×256 이미지 처리에 0.7초가 소요되어 차량 탑재 실시간 적용이 원천 불가(A003)
  ```

### S02. 서술식 (논문/보고서)

- **용도**: 연구보고서, 기술보고서, 백서, 학술 제안
- **구조**: 문단 중심, 제목-본문-결론 흐름
- **어미**: "~이다", "~한다", "~있다" (정중체)
- **원칙**: 논리적 흐름 (배경→문제→해결→기대효과), 정량 근거 인용, 참고문헌 표기
- **예시**:
  ```
  본 사업은 사내 코드베이스에 특화된 AI 코딩 어시스턴트를 개발하는 것을 목적으로 한다.
  기존 벡터 검색 방식의 한계를 극복하기 위해 GraphRAG 기반 의미론적 코드 검색 기술을
  적용하여 코드 검색 정확도를 향상시킨다.
  ```

### S03. 개조식 2단 (간결)

- **용도**: 요약서, 발표자료 원고, 견적 설명, 짧은 제안
- **구조**:
  - 1단(■): 항목명
  - 2단(-): 세부 내용
- **어미**: "~임", "~완료", "~예정" (명사형 종결)
- **원칙**: 항목당 2줄 이내, 핵심만 기술
- **예시**:
  ```
  ■ AI 코딩 어시스턴트 개발
    - GraphRAG 기반 코드 검색, 정확도 30% 향상 목표
    - 개발 기간: 6개월, 투입 인력: 3명
  ```

### S04. 혼합식 (서술+개조)

- **용도**: 민간 제안서, 사업계획서, 투자 제안, 기획서
- **구조**: 도입부 서술(2~3문장) + 핵심 내용 개조식
- **어미**: 서술부 "~합니다" (경어체), 개조부 "~임", "~예정"
- **원칙**: 도입부에서 맥락 설명, 개조식에서 핵심 전달
- **예시**:
  ```
  사내 코드베이스의 규모가 증가함에 따라 개발자의 코드 검색 및 재사용에
  소요되는 시간이 전체 개발 시간의 30%를 차지하고 있습니다.
  이를 해결하기 위해 다음과 같은 AI 코딩 어시스턴트를 개발합니다.

  □ 핵심 기능
    ○ GraphRAG 기반 의미론적 코드 검색
      - 기존 벡터 검색 대비 정확도 30% 향상
    ○ 코딩 컨벤션 자동 적용
      - 사내 아키텍처 패턴 학습 기반
  ```

## HWP 서식 분석 가이드

> HWP/HWPX 양식 문서를 받았을 때 **반드시 서식(타이포그래피)도 분석**하여 `12_양식분석.md`에 기록한다.
> 내용 구조 분석(S01~S04)과 별도로, 폰트·단락 서식은 산출물을 HWP로 복원할 때 필수 정보다.

### 분석 항목

| 항목 | 확인 내용 |
|------|---------|
| 폰트명 | KoPub, 나눔, 맑은 고딕 등 |
| 폰트 굵기 | Bold / Light / Regular |
| 크기 | pt 단위 (소수점 포함 가능: 11.5pt) |
| 줄간격 | % 또는 pt (예: 160%) |
| 장평 | % (예: 100%) |
| 자간 | 숫자 (예: 0, -5) |
| 적용 위치 | 어느 스타일(제목/본문 등)에 해당하는지 |

### 분석 절차 (양식 문서 수령 시)

1. HWP/HWPX → MD 변환 (`oowork extract` 또는 `hwp_to_md.py`)
2. 원본 HWP를 한글 프로그램에서 열어 **문단 스타일** 목록 확인
3. 각 스타일별 폰트·크기·줄간격·장평·자간을 표로 정리
4. `{작업폴더}/01_RFP/12_양식분석.md` §서식 스타일 섹션에 기록
5. 이후 산출물 작성 시 해당 서식을 준수

## 스타일 등록 형식

새 스타일을 등록할 때 아래 형식을 따른다:

```markdown
### S{번호}. {스타일명}

- **용도**: {적합한 문서 유형}
- **구조**: {계층 구조 또는 문단 구조 설명}
- **어미**: {종결 어미 규칙}
- **원칙**: {핵심 작성 원칙}
- **예시**:
  ```
  {실제 예시 텍스트}
  ```
```

번호(S{N})는 기존 최대 번호 + 1로 할당한다.
