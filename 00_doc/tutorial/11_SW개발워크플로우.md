## 1. 개요

**이럴 때 사용:** 새 기능을 기획부터 커밋까지 체계적으로 완료할 때  
**결과물:** 구현 코드 + 테스트 + PRD/Plan/상세 문서 + Git 커밋

### 1.1. 실행 방식 선택

| 방식          | 명령어                        | 특징                           |
| ----------- | -------------------------- | ---------------------------- |
| **자동 (권장)** | `ooflow run`               | 완전 무인 자동화 — 사용자 확인 없이 끝까지 실행 |
| 단계별 확인      | `ooflow run --interactive` | 각 단계마다 사용자 승인 후 진행           |
| 특정 단계까지     | `ooflow run --until 검증`    | 지정 단계까지만 실행                  |
| 수동          | 아래 2~7절 개별 명령어             | 각 스킬을 직접 순서대로 실행             |

## 1.2. ooflow 자동 실행 (권장)

```bash
oostart run        # 세션 시작
oocontext 4        # SP 설정 (SP 번호는 상황에 맞게)
ooflow run         # 실행 계획 미리보기 → 확인 → 완전 무인 자동 실행
```

**ooflow run 실행 순서:**
```
[0] dry-run 자동 실행 → 실행 계획 간략 출력
        ↓
"위 계획으로 실행하시겠습니까? [y/N]"
        ↓ y
[1] PRD 확인/정합성 검증 (ooprd run)
[2] Plan 확인/갱신 (ooplan run)
[3] 미착수 Feature 목록 추출 (oofeature needed)
[4] Feature별 자동 반복 (이후 사용자 확인 없음):
    ├─ 상세기획 생성 (oofeature next)
    ├─ 기획→설계 전환 (oofeature next)
    ├─ 설계→개발 TDD (oodev run dXXXX)
    ├─ 개발→검증 체크 (oocheck run dXXXX)         ← 정적 분석
    ├─ 이슈 자동 수정 (oofix run — oocheck CRITICAL/ERROR)
    ├─ AI 리뷰 (ooreview run dXXXX)               ← 설계·보안·품질 + Codex 2차 (기본 ON)
    ├─ 이슈 자동 수정 (oofix run — ooreview CRITICAL/ERROR, 최대 2회)
    ├─ 완료 처리 (oofeature next)
    └─ Feature 커밋 (oocommit run)
[5] 문서 일괄 업데이트 (oodoc run)
[6] 이슈 아카이브 (oohistory run)
[7] 최종 커밋 (oocommit run)
```

**이슈 자동 처리 규칙:**

| 심각도 | 처리 |
|--------|------|
| CRITICAL | oofix 자동 실행 → 재검증 (최대 3회) |
| ERROR | oofix 자동 실행 → 재검증 1회 |
| WARNING | todo 등록 후 계속 진행 |

**강제 중단 조건:** CRITICAL 3회 반복 실패 | 컨텍스트 85% 초과 | 파일 시스템 오류

```bash
ooflow run --yes            # 확인 생략, 즉시 자동 실행
ooflow run --dry-run        # 실행 계획만 출력 (실행 안 함)
ooflow run --interactive    # 단계별 사용자 확인
ooflow run --feature d41001 # 특정 Feature 1개만 자동 처리
ooflow run --no-review      # ooreview 단계 생략 (빠른 실행, 모든 Feature 적용)
ooflow status               # 현재 SP 워크플로우 진행 현황
```

```

→ [Feature별 반복]

    oofeature next dXXXX            ← 완료 처리
    oocommit run
→ oodoc run → oohistory run → oocommit run → oostop run
```

## 3. 요구사항 정의 (PRD)

> **PRD 역할**: "무엇을(What)" 정의 — 3번 구현 계획에서 Feature 분해의 근거가 됨

```bash
ooprd run          # PRD 생성 → d{SP}0001_prd.md
```

**프로젝트 유형별 템플릿 선택:**

| 템플릿 | 유형 | 적용 상황 |
|--------|------|----------|
| common | 범용 | 일반 프로젝트 |
| streamlit | Type B | 웹 앱 (Streamlit) |
| algorithm | Type A | 알고리즘/분석/ML |
| agent | Type C | 에이전트/CLI 도구 |

```bash
ooprd run --template streamlit   # 템플릿 지정 생성
```

**ooplan Feature 분해를 위해 PRD에 반드시 포함해야 할 항목:**

| PRD 섹션 | 내용 | ooplan 활용 |
|----------|------|------------|
| §기능요구사항 | MoSCoW 기능 목록 | Epic → Feature 분해 근거 |
| §기능 상세 | 기능별 입출력/시나리오 | Feature → Task 세분화 근거 |
| §기술스택 | 언어/프레임워크 요약 | 기술 설계 방향 |

> PRD §기능요구사항이 불충분하면 ooplan이 Feature를 제대로 분해할 수 없음

**모호한 요구사항 해소 (Plan 전에 필수):**
```bash
ooprd clarify      # 최대 5개 질문으로 모호성 해소 → PRD 자동 업데이트
```

**이미 PRD가 있으면:**
```bash
ooprd run          # 기존 PRD 정합성 검증 (자동 분기)
ooprd validate     # 구조 검증 (필수 섹션 누락 여부)
```

## 4. 구현 계획

```bash
ooplan run         # PRD 기반 계획 생성 → d{SP}0002_plan.md
```

**Feature 목록 생성 기준 — PRD 자동 분해:**

```
PRD(무엇을) → Epic(도메인) → Feature(사용자 기능) → Task(구현 단위)
```

- PRD §기능요구사항이 Feature 목록의 근거
- `ooplan run` = PRD를 읽어 Epic > Feature > Task 자동 분해
- PRD §기능 상세가 충분해야 Feature 단위까지 분해 가능

**생성되는 내용:**
- Phase별 구현 순서 (Phase 1: 기반 → Phase 2: 핵심 → Phase 3: 고도화)
- 파일/모듈 목록 및 의존성
- Feature/Task 구조 (8.2절: 상세 문서 현황)

**Plan 완료 후 — 미착수 Feature 확인:**

```bash
oofeature needed   # plan.md Feature vs 상세 문서 교차 비교 → 미착수 목록 출력
```

```
[oofeature needed] SP04 — 상세기획 미착수 Feature

| Feature ID | 기능명          | Epic | 우선순위 | 권장 시작 단계 |
|-----------|----------------|------|:-------:|:------------|
| F001-1    | 데이터수집소스   | E001 | high    | 상세기획     |
| F001-2    | 크롤러 모듈 설계 | E001 | high    | 상세기획     |

총 2개 미착수 (전체 7개 중)
```

> 상세 문서(기획/설계/개발/검증 어떤 단계든)가 하나라도 존재하면 "착수"로 간주

### 5.3. 설계 → 개발 (TDD)

```bash

# → 설계 단계 감지 시 파일명 자동전환 (상세설계 → 상세개발)
```

> ⚠️ `oodev run dXXXX`가 TC 작성(RED)~REFACTOR까지 통합 수행. 별도 `ootest write` 불필요.

**`oodev run dXXXX` 동작:**
1. `00_doc/` 하위에서 `d41001_*.md` 자동 탐색
2. 상세설계 단계이면 → 파일명 rename (상세설계 → 상세개발) 자동 수행
3. 문서에서 Feature/Task 추출 (없으면 AI 자동 분해)
4. TC 코드 작성(RED) → GREEN(최소 구현) → REFACTOR → VERIFY(런타임 검증)
5. 완료 상태를 `d{SP}0003_test.md`에 기록

## Plan 변경 시 대응

상세개발·검증 진행 중 `plan.md`가 수정된 경우:

```bash
oofeature update --from-plan                    # 영향 문서 분석 (롤백 후보 표시)
oofeature update --from-plan --apply            # 실제 롤백 실행 (이전 단계로 rename)
oofeature update --from-plan --apply --dry-run  # 미리보기 (실제 수정 안 함)
```

**동작 방식:**
1. `git diff HEAD` 기반으로 plan.md 변경 감지
2. 변경된 줄에서 doc 번호(`d41001` 등) 추출
3. 해당 번호의 상세 문서(설계/개발/검증 단계) 식별
4. 이전 단계로 rollback 제안 (개발→설계, 검증→개발 등)

**diff 없는 경우:** plan.md 미수정으로 감지되면 전체 활성(설계/개발/검증) 문서 목록 표시

| 옵션 | 설명 |
|------|------|
| (없음) | 영향 문서 분석 + 롤백 후보 출력만 |
| `--apply` | 실제 파일 rename (이전 단계로 롤백) |
| `--apply --dry-run` | rename 미리보기 (실제 수정 안 함) |
| `--sp N` | 특정 SP만 분석 |

## 6.5. ooflow 완료 후 권장 — 수동 Codex 리뷰 (선택)

> ooflow 자동화 종료 후 **변경분 전체**에 대해 Codex 인터랙티브 리뷰를 한 번 더 받는다. 자동 ooreview는 codex:rescue(read-only)로 의견만 받지만, 이 단계는 **Codex 슬래시 명령으로 풍부한 UI**에서 직접 검토한다.

**삽입 시점**: `oocommit run`(최종 커밋) 직후, `oostop run` 전

```
/codex:review              # 변경분 일반 리뷰 (설계·버그·개선)
/codex:adversarial-review  # 보안·설계 공격적 리뷰 (7개 공격 면)
```

### 6.5.1. 세 가지 리뷰 경로 비교

| 항목 | ooreview (자동) | /codex:review (수동) | /codex:adversarial-review (수동) |
|------|----------------|---------------------|--------------------------------|
| 호출 시점 | Feature별 검증마다 자동 | ooflow 종료 후 입력 | ooflow 종료 후 입력 |
| 호출 경로 | Task → codex:rescue (read-only) | 슬래시 (인터랙티브) | 슬래시 (인터랙티브) |
| 검토 범위 | Feature 단위 (좁고 깊게) | 변경분 전체 | 변경분 전체 |
| 검토 관점 | 설계·보안·품질 일반 | 일반 (버그·설계·개선) | **공격적 7개 면 (보안 특화)** |
| 결과 표현 | 텍스트 합산 → d{SP}0004 등록 | UI 인라인 인용·색상 | UI 인라인 + 공격 시나리오 |
| ooflow 통합 | ✅ 모든 Feature 자동 | ❌ 인터랙티브 | ❌ 인터랙티브 |
| 권장 시점 | 매 Feature | Phase/마일스톤 종료 | **보안 민감 코드 배포 전** |
| 비용 | Codex 사용량 차감 | Codex 사용량 차감 | Codex 사용량 차감 (가장 큼) |

### 6.5.2. /codex:review 사용

**용도**: 일반적인 코드 변경분 종합 리뷰. 설계 의도·잠재 버그·개선 제안 위주.

```bash
/codex:review --wait         # 작은 변경 (포어그라운드 대기)
/codex:review --background   # 큰 변경 (백그라운드 실행)
/codex:status                # 진행 상황 확인
/codex:result                # 결과 조회
```

**권장 시점**:
- Phase 내 모든 Feature 완료 후 종합 검토
- 마일스톤 종료 직전 코드 정리
- PR 작성 전 self-review

### 6.5.3. /codex:adversarial-review 사용 (보안 특화)

**용도**: 적대적 관점에서 **7개 공격 면**을 체계적으로 검토. `/codex:review`보다 보안·신뢰성에 특화.

```bash
/codex:adversarial-review              # 7개 공격 면 일괄 검토
/codex:adversarial-review --background # 큰 변경은 백그라운드
```

**7개 공격 면 상세:**

| # | 항목 | 검토 포인트 | 예시 시나리오 |
|---|------|----------|------------|
| 1 | 인증/권한 | 접근 제어 결함 | 권한 없는 사용자가 관리자 API 호출 |
| 2 | 데이터 손실 | 데이터 유실 가능성 | 트랜잭션 실패 시 부분 커밋, 백업 누락 |
| 3 | 롤백 안전성 | 배포 롤백 가능 여부 | 마이그레이션 일방향, 다운그레이드 불가 |
| 4 | 레이스 컨디션 | 동시성 문제 | 동시 쓰기 시 TOCTOU, 잠금 누락 |
| 5 | 빈 상태 | 엣지케이스 (빈 배열·null·0) | 첫 사용자, 빈 응답, 미초기화 상태 |
| 6 | 버전 스큐 | 구버전 클라이언트 호환성 | 신규 필드 추가 시 구버전 앱 크래시 |
| 7 | 관찰성 갭 | 로깅·모니터링 누락 | 오류 시 silent fail, 메트릭 미발행 |

**필수 적용 시점** (선택이 아닌 권장):
- 🔒 **인증·세션·토큰 처리** 변경 시
- 🔒 **결제·금융·개인정보** 처리 변경 시
- 🔒 **외부 API 노출** (REST/GraphQL endpoint 추가) 시
- 🔒 **DB 스키마 마이그레이션** 시
- 🔒 **롤아웃·배포 직전** (프로덕션 push 전)

### 6.5.4. 사용 가이드라인

| 상황 | 권장 명령 |
|------|----------|
| 일반 기능 추가 | `/codex:review` |
| 리팩토링/내부 정리 | `/codex:review` |
| 보안 민감 기능 (auth, payment, PII) | **`/codex:adversarial-review`** |
| 외부 노출 API | **`/codex:adversarial-review`** |
| DB 스키마 변경 | **`/codex:adversarial-review`** |
| Phase/마일스톤 종료 | 둘 다 (`review` → `adversarial-review`) |
| 프로덕션 배포 직전 | 둘 다 + 보안 영역 집중 검토 |

> 💡 비용: Codex 사용량 별도 차감. `adversarial-review`는 `review`보다 토큰 소모가 큼.  
> 💡 백그라운드 권장: 큰 변경분은 `--background` + `/codex:result`로 비동기 처리.

### B1. 코드 리뷰 (ooreview) — 메인 흐름 통합됨 (v05)

> ✅ **메인 흐름에 통합 완료** (2026-05-14, v05). 5.5절 참조. ooflow는 **모든 Feature에 ooreview 자동 호출**한다.

**ooflow에서의 동작:**

```
oocheck run dXXXX  →  ooreview run dXXXX  →  oofeature next dXXXX  →  oocommit run
   (정적 분석)         (AI 리뷰: 설계+보안+품질+Codex 2차)
```

**리뷰 생략 옵션 (모든 Feature 일괄):**

```bash
ooflow run --no-review        # ooreview 단계 전체 생략 (빠른 실행)
```

> ⚠️ `--no-review`는 **모든 Feature**에 일괄 적용된다. Feature별 선택적 생략은 지원하지 않는다 (정책: 일관성 우선).

**수동 실행 시 단독 사용:**

```bash
ooreview run d41001         # 특정 Feature만 리뷰
ooreview security           # 보안 집중 리뷰
ooreview codex [파일]       # Codex 단독 2차 리뷰
```

**리뷰 영역별 담당 에이전트:**

| 영역 | 에이전트 | 모델 |
|------|---------|------|
| 설계·패턴 | code-reviewer | opus |
| 보안 (OWASP Top10) | security-reviewer | sonnet |
| 품질·복잡도 | quality-reviewer | sonnet |
| 엣지케이스·로직 결함 (2차 의견) | codex:rescue (read-only) | - |

### B2. 통합 테스트 (여러 Feature 완료 후)

개별 Feature 검증(oocheck)은 단위 수준이며, **여러 Feature 완료 후 전체 통합 테스트**가 필요합니다.

**삽입 시점:** Phase 내 모든 Feature 완료 후, `oodoc run` 전

```bash
# Phase 1 Feature 전부 완료 후
ootest run              # 전체 TC 실행 + 커버리지 확인
ootest coverage         # 커버리지 리포트 (목표: 80%+)

# 실패 시
oofix run               # 회귀 오류 수정
ootest run              # 재실행 확인
```

**회귀 방지 체크 (Feature 추가할 때마다):**
```bash
ootest run --regression   # 기존 TC가 새 구현으로 깨지지 않는지 확인
```

### B3. 컨텍스트 초과 시 세션 재개

```bash
# 1. 현재 진행 상태 확인
```

```bash
ootoken                 # 현재 토큰 사용량 확인

```

### B4. Feature 간 의존성 처리

Feature A가 Feature B의 코드에 의존할 때 순서 관리가 필요합니다.

**원칙:** `plan.md`의 Phase 순서 = 구현 순서 (낮은 Phase 먼저)

**의존 관계 확인:**
```bash
oofeature list          # 현재 단계 확인
# plan.md 8.2절에서 의존 관계 수동 확인
```

**의존 Feature가 완료되지 않은 경우:**
```bash
# 의존 Feature 먼저 처리
oofeature next d41001   # 선행 Feature 완료
oocommit run

# 그 다음 의존하는 Feature 진행
oofeature next d41002
oodev run d41002
```

> `oodev run d41002` 실행 전 선행 Feature 코드가 없으면 TC가 import 오류로 실패함 — 반드시 선행 완료 후 진행

### B5. 패키지 의존성 추가 (oouv)

개발 중 새 외부 패키지가 필요한 경우 `pyproject.toml` 갱신이 워크플로우에 포함되지 않습니다.

**삽입 시점:** `oodev run` 중 import 오류 발생 시

```bash
# 패키지 추가
oouv add requests           # pyproject.toml에 추가 + uv sync

# 추가 후 반드시
ooenv uv check              # 충돌/취약점 확인
oocommit run                # pyproject.toml, uv.lock 함께 커밋
```

**패키지 추가 커밋 메시지 형식:**
```
chore(deps): requests 추가 — SP04 웹 크롤링 기능
```

### B6. 실행 확인 (배포/동작 검증)

구현이 완료됐더라도 **실제 실행 동작**을 확인하는 단계가 없습니다.

**삽입 시점:** `ootest run` 통과 후, 최종 `oocommit run` 전

```bash
oorun run               # 프로젝트 실행 (설정된 실행 방법으로)
# → 예: uv run python main.py, streamlit run app.py 등

# 실행 오류 발생 시
oocheck run             # 런타임 에러 재분석
oofix run               # 수정
oorun run               # 재확인
```

**SP 유형별 실행 확인:**

| SP 유형 | 확인 방법 |
|---------|----------|
| Streamlit 앱 | `streamlit run app.py` → UI 동작 확인 |
| Python 스크립트 | `uv run python main.py` → stdout/stderr 확인 |
| API 서버 | `uvicorn main:app` → 주요 엔드포인트 curl 테스트 |
| CLI 도구 | `uv run python cli.py --help` → 주요 명령어 실행 |
