# 개발 시나리오: 모듈/라이브러리 수정

> oo 공통 모듈 수정 시 영향 분석부터 검증까지의 흐름을 안내합니다.

---

## 개요

oo 공통 모듈(여러 SP에서 공유하는 라이브러리)을 수정할 때의 흐름입니다. 수정 전 영향 분석으로 어떤 파일이 영향받는지 파악하고, 수정 후 회귀 테스트로 기존 기능이 깨지지 않았는지 검증합니다.

**이럴 때 사용:** oo/ 공통 함수/클래스를 변경할 때, 여러 SP에 영향이 있는 수정 시
**결과물:** 수정된 모듈 + 회귀 테스트 통과 + d0005_lib.md 업데이트


## 전체 흐름

```
oolib 영향 분석 → 수정 → oocheck 전체 검증 → ootest 회귀 테스트 → oocommit
```

---

## 1. 영향 분석

```bash
oolib run          # oo 모듈 분석 + 수정
```

**분석 항목:**
- 수정 대상 함수/클래스 식별
- 해당 함수를 참조하는 모든 파일 목록
- 하위 호환성 영향 평가

**예시:** `oo/metrics.py`의 `seg_acc()` 수정 시
```
참조 파일:
  - 02_pycode/ps10_seg/train.py (line 45)
  - 02_pycode/ps10_seg/eval.py (line 78)
  - tests/test_metrics.py (line 12, 34)
```

---

## 2. 수정

### 수정 원칙
- **하위 호환성 유지**: 기존 호출부가 깨지지 않도록
- **시그니처 변경 시**: deprecated 경고 추가 → 다음 버전에서 제거
- **새 기능 추가 시**: 기존 동작에 영향 없는지 확인

### SP 컨텍스트 주의
```bash
oocontext clear    # 공통 모듈 수정 시 SP00으로 전환
```
- 공통 모듈 이슈는 `d0004_todo.md`에 등록 (SP별 아님)

---

## 3. 전체 검증

```bash
oocheck run        # 수정으로 인한 새 이슈 발생 여부 체크
```

**체크 대상:**
- 수정된 모듈 자체
- 수정된 모듈을 참조하는 모든 코드
- import 에러, 타입 불일치, 미사용 변수

---

## 4. 회귀 테스트

```bash
ootest run         # 기존 테스트 전체 실행
```

**목표:** 기존 테스트 100% 통과 (수정으로 인한 회귀 없음)

실패하는 테스트가 있으면:
- 의도된 변경 → 테스트 수정
- 의도하지 않은 변경 → 코드 수정

---

## 5. 문서 + 커밋

```bash
oodoc run          # d0005_lib.md 업데이트
oocommit run       # refactor(oo): 모듈 변경 내용 요약
```

---

## FAQ

**Q: oo 모듈과 SP 코드를 동시에 수정해야 하면?**
A: oo 모듈 먼저 수정 (SP00) → SP 코드 수정 (`oocontext N`) → 각각 커밋

**Q: 여러 SP에서 참조하는 함수를 변경하면?**
A: `oocheck run`이 모든 SP의 영향을 분석합니다. SP별로 확인 후 수정.
