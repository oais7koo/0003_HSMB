# oof 튜토리얼 (oofeature 별칭)

## § 1. 필요한 이유

기능 개발 시 계획, 설계, 개발, 검증 단계마다 문서를 따로 관리하는 것은 혼란스럽습니다. 단 하나의 파일명으로 현재 단계를 즉시 파악할 수 있고, 단계별 표준 문서 구조를 자동 제공하면 팀 전체가 일관된 품질로 개발할 수 있습니다. 이것이 oof(oofeature)의 목표입니다.

## § 2. 빠른 시작 (3가지 기본 명령)

```bash
# 1. 상세기획 문서 생성
oof new d41001 "데이터수집소스"

# 2. 다음 단계로 전환 (기획 → 설계 → 개발 → 검증 → 완료)
oof next d41001

# 3. 미착수 기능 확인
oof needed
```

## § 3. 자주 쓰는 명령어

| 명령어 | 사용 시점 | 설명 |
|--------|---------|------|
| `oof list` | 작업 시작 전 | 현재 SP의 상세 문서 목록 + 단계 |
| `oof needed` | 구현 계획 수립 시 | plan.md에서 상세기획 미착수 기능 리스트업 |
| `oof new d{} "기능명"` | 신규 기능 시작 | 상세기획 문서 자동 생성 |
| `oof next d{}` | 각 단계 완료 후 | 다음 단계로 파일 rename + 연계 스킬 자동 실행 |
| `oof note d{} "내용"` | 기록 필요 시 | 상세 문서에 날짜+내용 추가 |

## § 4. 권장 워크플로우

```
1. [기능 목록 확인]
   oof needed → 미착수 기능 확인
   또는
   oof list → 현재 진행 상황 확인

2. [기획 단계]
   oof new d41001 "기능명"
   → d41001_상세기획_기능명.md 생성
   → 기획 내용 작성

3. [설계 단계]
   oof next d41001
   → d41001_상세설계_기능명.md 로 rename
   → 설계 내용 작성 후 재실행

4. [개발 단계]
   oof next d41001
   → d41001_상세개발_기능명.md 로 rename
   → oodev run d41001 자동 연계 실행 (TDD 개발)

5. [검증 단계]
   oof next d41001
   → d41001_상세검증_기능명.md 로 rename
   → oocheck run d41001 자동 연계 실행 (코드 검증)

6. [완료 단계]
   oof next d41001
   → d41001_상세완료_기능명.md 로 rename
   → oocommit run 자동 연계 실행
```

## § 6. 상세 사용 방법

### 6.1 파일명 규칙

```
d{SP}{기능번호}_상세{단계}_{기능명}.md
```

**단계 순서**:
```
⚪ 상세기획 → 🔵 상세설계 → 🟡 상세개발 → 🟢 상세검증 → ✅ 상세완료
```

**예시**:
```
d41001_상세기획_데이터수집소스.md    ← 기획 단계
d41001_상세설계_데이터수집소스.md    ← 설계 단계
d41001_상세개발_데이터수집소스.md    ← 개발 단계
d41001_상세검증_데이터수집소스.md    ← 검증 단계
d41001_상세완료_데이터수집소스.md    ← 완료 단계
```

**번호 규칙**:
- `d{SP}1000~9999`: 상세 문서 전용 범위
- 큰 기능별로 1000, 2000, 3000... 단위 할당
- 세부 번호는 10 단위 사용 (1010, 1020, 1030...)

### 6.2 문서 템플릿

`oof new`로 생성되는 문서는 자동으로 적절한 템플릿 선택:

| 조건 | 템플릿 | 선택 방법 |
|------|--------|---------|
| SP04(Streamlit 백오피스) | `상세기획_template_streamlit.md` | 자동 감지 또는 `--framework streamlit` |
| SP05(FastAPI) | `상세기획_template_fastapi.md` | 자동 감지 또는 `--framework fastapi` |
| 그 외 | `상세기획_template.md` | 기본값 |

**템플릿 명시 지정**:
```bash
oof new d41001 "기능명" --framework streamlit
oof new d51001 "기능명" --framework fastapi
```

### 6.3 단계별 문서 구조

**기획 단계** (`상세기획_*.md`):
- 1. 기능 개요
- 2. 요구사항
- 3. 입출력 정의
- 4. 제약조건 / 예외처리
- 5. 관련 Feature
- 6. 참고 자료

**설계 단계** (기획 문서에 추가):
- A. 설계 상세 (아키텍처/API/DB)

**개발 단계** (설계 문서에 추가):
- B. 구현 노트 (Task 체크리스트)

**검증 단계** (개발 문서에 추가):
- C. 검증 결과 (TC 통과 현황)

**완료 단계** (검증 문서에 추가):
- D. 완료 확인 (검증 통과 확정·서명)

## § 7. 실전 시나리오

### 7.1 기능 1개 완전히 처리

```bash
# 1. 기획 문서 생성
oof new d41001 "사용자인증시스템"

# 2. 기획 작성 (d41001_상세기획_사용자인증시스템.md 편집)
# vim 또는 에디터에서 기획 내용 작성

# 3. 기획 → 설계 전환
oof next d41001
# → d41001_상세설계_사용자인증시스템.md 로 rename
# → 설계 작성 후 재실행

# 4. 설계 → 개발 전환
oof next d41001
# → d41001_상세개발_사용자인증시스템.md 로 rename
# → oodev run d41001 자동 실행

# 5. 개발 → 검증 전환
oof next d41001
# → d41001_상세검증_사용자인증시스템.md 로 rename
# → oocheck run d41001 자동 실행

# 6. 검증 → 완료 전환
oof next d41001
# → d41001_상세완료_사용자인증시스템.md 로 rename
# → oocommit run 자동 실행
```

### 7.2 미착수 기능 찾아서 할당

```bash
# 1. 현재 상황 확인
oof list

# 2. 미착수 기능 확인
oof needed
# 출력:
# [oof needed] SP04 — 상세기획 미착수 Feature
# | F001-1 | 사용자인증 | E001 | high | 상세기획 |
# | F001-2 | 권한관리 | E001 | high | 상세기획 |

# 3. 미착수 기능부터 시작
oof new d41002 "권한관리"
oof next d41002
...
```

### 7.3 문서에 메모/이슈 추가

```bash
# 메모 추가 (타임스탐프 자동 기록)
oof note d41001 "API 엔드포인트 변경 필요"
oof note d41001 "입력 유효성 조건 추가"

# 이슈 추가 (🔴 미해결 상태)
oof issue d41001 "DB 스키마 아직 미정의"
oof issue d41001 "외부 API 의존성 확인 필요"

# 이슈 해결 처리
oof issue d41001 --resolve
# → 가장 최신의 미해결 이슈를 ✅ 해결 로 변경
```

## § 9. FAQ

**Q. oof와 oofeature의 관계는 무엇인가요?**
A. oof는 oofeature의 별칭(alias)입니다. `oof next d41001` = `oofeature next d41001`

**Q. 파일을 실수로 삭제했습니다. 복구 방법이 있나요?**
A. git 이력에서 복구할 수 있습니다. `git checkout HEAD~1 -- 파일명`

**Q. 여러 SP에서 동시에 작업할 수 있나요?**
A. `oocontext {SP번호}`로 SP를 전환한 후 해당 SP의 문서를 생성하세요.

**Q. 기존 파일(d40xxx)과 새 번호(1000번대)를 섞어서 사용할 수 있나요?**
A. 신규 문서는 1000번대부터 사용하세요. 기존 d40xxx는 그대로 유지 가능합니다.

**Q. next를 실수로 실행했습니다.**
A. 파일명을 수동으로 원래 단계로 변경하거나 깃 이력에서 복구하세요.
