## 1. 개요

이 문서는 **Claude 환경으로 세팅된 OAIS 프로젝트에서 Copilot CLI가 어디까지 활용 가능한지, 그리고 왜 일부는 활용할 수 없는지**를 설명한다.  
기준은 단순한 “기능 유무”가 아니라, **현재 프로젝트의 파일/문서/스킬 구조를 Copilot가 실제로 읽고 실행할 수 있는가**이다.

Copilot CLI는 **터미널에서 동작하는 별도 에이전트**로, 현재 작업 디렉터리의 코드와 허용된 파일/도구를 중심으로 동작한다.

주로 활용하는 환경은 다음과 같다.

- 저장소의 지시 파일 (에이전트마다 읽는 파일이 다름)

| 에이전트 | 주 지시 파일 |
|----------|-------------|
| Claude Code | `CLAUDE.md` |
| Copilot CLI | `AGENTS.md`, `.github/copilot-instructions.md` |
| Gemini | `AGENTS.md` |

- 현재 폴더의 코드와 문서
- 선택적으로 IDE, LSP, MCP, GitHub 이슈/PR 컨텍스트

즉, **Claude 환경이 이미 세팅되어 있어도 Copilot가 그 전체를 자동으로 다 쓰는 것은 아니고**, 읽을 수 있는 범위와 실행 권한이 겹치는 부분만 활용한다.  
활용이 안 되는 부분은 보통 **도구 이름/에이전트가 다르거나, 세션 상태를 공유하지 않거나, 파일/권한/실행 정책이 다르기 때문**이다.  
또한 프롬프트 1회마다 **premium request 1회**가 차감된다.

여기서 **프롬프트 1회**는 Copilot CLI에 한 번 질문하거나 작업을 요청해 응답을 받는 **단일 대화/실행 단위**를 뜻한다.


## 2. Claude Code와의 호환성

Copilot CLI는 Claude Code와 비슷한 대화형 개발 흐름을 따르지만, **동일한 도구도 아니고 동일한 실행 규칙도 아니다**.  
따라서 이 장의 핵심은 “비슷하다/다르다”가 아니라, **어디까지는 활용 가능하고 왜 그 이상은 활용되지 않는가**다.

최근 oo 스킬들이 `스킬 요약`을 추가하면서, 예전보다 훨씬 더 명확하게 **무엇을 하고 / 무엇을 하지 않는지 / 어디를 참조하는지 / 자동인지 반자동인지**가 드러나게 되었다.  
그래서 Copilot가 활용 가능한 부분과 불가능한 부분을 구분할 때도, 이제는 각 스킬의 `참조 범위`와 `하지 않는 것`을 기준으로 보면 된다.

### 2.1 공통적으로 동작하는 부분

- 자연어로 코드 작업과 문서 작업을 요청할 수 있다.
- 현재 프로젝트의 파일과 문서를 기준으로 작업한다.
- 계획 → 구현 → 검증 → 정리의 순서를 따라갈 수 있다.
- 지시 문서와 프로젝트 규칙을 함께 반영한다.

### 2.2 최근 oo 스킬에서 명확해진 공통 구조

현재 oo 스킬들은 대부분 아래 7가지를 공통으로 보여준다.

1. **핵심 역할**
2. **하는 것**
3. **하지 않는 것**
4. **참조 범위**
5. **수정 대상**
6. **실행 레벨**
7. **에이전트 호환**

이 구조 덕분에 예전처럼 “이 스킬이 어디까지 책임지는지”를 추측할 필요가 많이 줄었다.  
특히 `oostart`, `oocontext`, `ooflow`, `oodev`, `oocheck`, `oosync`, `oostop`, `ooskill`이 각각 자기 역할을 더 선명하게 분리했다.

### 2.3 스킬별로 다르게 이해해야 하는 부분

> 세션 생명주기 순서: `oostart → oocontext → ooflow → oodev → oocheck → oosync → oostop → ooskill`

#### 1) `oostart`는 시작 점검만 하고 끝난다

`oostart`는 세션 시작 시:

- 서브프로젝트 스캔
- common guide 로드
- 문서 상태 점검
- oocheck 연동 안내

를 수행하지만, **코드를 수정하지 않는다.**

즉, 시작 전에 “지금 작업할 준비가 되었는지” 보는 스킬이지, 개발 스킬이 아니다.  
Copilot CLI에서 세션 초기에 보는 환경 점검과는 유사하지만, `oostart`는 **문서 기반 점검 절차가 매우 고정적**이다.

#### 2) `oocontext`는 세션 상태만 바꾼다

`oocontext`는 파일을 수정하지 않고, **현재 세션의 SP 컨텍스트만 설정**한다.  
즉, 실제 코드나 문서를 바꾸는 스킬이 아니라 **이후 스킬들이 어떤 SP 기준으로 해석할지 정하는 스위치**다.

이 스킬을 Copilot CLI 관점으로 보면 다음 차이가 있다.

- SP는 **세션 컨텍스트** 중심이고,
- 파일 시스템 자체를 바꾸지 않으며,
- 외부 프로젝트를 자동 연결하지 않는다.

그래서 “현재 프로젝트”를 자동 추론하는 기능과는 비슷해 보여도, 실제로는 **문서/문맥 기준을 고정하는 역할**에 가깝다.

#### 3) `ooflow`는 전체 오케스트레이션이지만 직접 구현은 아니다

`ooflow`는 전체 SW 개발 흐름을 묶는 스킬이지만, **실제 구현을 직접 하지 않는다.**  
지금 요약상 핵심은 다음이다.

- `oofeature → oodev → ootest → oocheck → ooreview → oofix → oodoc → oocommit`
- `--dry-run`으로 미리보기
- `--yes`로 즉시 자동 실행
- `--interactive`로 단계별 확인
- `--ralph`로 실패 에스컬레이션

즉, `ooflow`는 **자동화 조정자**이지 작업자 자체가 아니다.  
Copilot CLI에서 비슷한 오케스트레이션을 기대할 수는 있어도, `ooflow`가 전제하는 단계와 정책은 훨씬 더 세밀하다.

#### 4) `oodev`는 현재 프로젝트 내부 구현 전용이다

`oodev`는 TDD의 **GREEN/REFACTOR**를 담당하고, 참조 범위도 `plan.md`, `test.md`, 코드 등 **현재 프로젝트 내부 파일만**으로 제한된다.  
또한 에이전트 흐름도 `explore → executor → reviewer`로 명시된다.

이 말은 곧:

- 다른 프로젝트 파일은 자동 포함하지 않고,
- TC 작성(RED)은 `ootest`가 맡고,
- 정적 분석은 `oocheck`가 맡는다

는 뜻이다.

Copilot CLI에서 기대하는 “한 번에 다 해주는 개발”과 비슷하지만, `oodev`는 **역할 분담이 더 강하게 고정된 개발 스킬**이다.

#### 5) `oocheck`는 읽기 전용 품질 검사다

`oocheck`는 현재 프로젝트의 `src/`, `oo/`, `tests/`, `lib/` 등을 검사하지만, **코드를 고치지 않는다**.  
발견한 이슈는 `d{SP}0004_todo.md`에 등록만 한다.

여기서 중요한 차이는:

- 검사 대상은 현재 프로젝트 내부로 제한되고
- 결과는 수정이 아니라 **이슈 기록**
- 실제 수정은 `oofix`의 영역

이라는 점이다.

즉, `oocheck`는 “문제 찾기”이고, “문제 해결”은 아니다.

#### 6) `oosync`는 동기화/백업 전용이다

`oosync`는 현재 프로젝트와 `OAIS_SYNC_TARGET` 경로 사이의 **환경 파일/설정 파일 동기화**를 다룬다.  
최근 요약을 보면 범위가 꽤 분명하다.

- 하는 것: `.claude/` 관련 파일 비교·복사, backup, restore, FTP 동기화
- 하지 않는 것: 코드 수정, Git 커밋, 환경 점검
- 참조 범위: 현재 프로젝트 + 대상 경로
- 실행 레벨: 반자동

여기서 Copilot CLI와 헷갈리기 쉬운 점은, `oosync`가 “프로젝트 전체를 복제”하는 게 아니라 **정해진 설정/문서/환경 파일만 맞추는 도구**라는 것이다.

#### 7) `oostop`는 종료 정리 전용이다

`oostop`는:

- 미완료 항목 정리
- README / doc 동기화
- 커밋 유도

를 담당하지만, **git push는 하지 않는다**.  
또한 실행 레벨이 반자동이라서, 종료 전에 정리할 항목을 보여주고 흐름을 마무리하는 성격이 강하다.

즉, `oostop`은 “세션 끝내기”이고 “배포 완료”는 아니다.

#### 8) `ooskill`은 스킬 검증 전용 메타 스킬이다

`ooskill`은 oo 스킬들을 직접 수행하는 게 아니라,  
`.claude/skills/oo*/SKILL.md`가 서브에이전트/명령어 활용을 잘 하고 있는지 **검증하고 개선하는 스킬**이다.

특히 다음을 본다.

- `help`는 스크립트로 파싱되는가
- `validate-checklist` 포맷이 맞는가
- `version` 규칙이 맞는가
- 스킬 구조가 표준을 따르는가

이건 Copilot CLI의 일반 작업보다는 **메타 품질 관리**에 가깝다.

### 2.4 아직 Copilot CLI가 활용하지 못하는 부분과 이유

아래는 “안 된다”가 아니라 **왜 그대로는 활용되지 않는지**를 적은 목록이다.

| 활용 불가/제한 대상 | 이유 |
|---|---|
| `oocontext`의 SP 기반 세션 전환 | Copilot CLI는 OAIS의 SP 컨텍스트를 자동으로 공유하지 않음. SP는 OAIS 세션 규칙이고 Copilot의 기본 세션 상태가 아님 |
| `d{SP}0004` 같은 문서 번호 체계 | 파일명 규칙이 OAIS 전용이라, Copilot가 범용 규칙처럼 해석할 근거가 없음 |
| `task-executor`, `task-checker`, `ooqa` 같은 내부 에이전트 호출 | Copilot CLI 기본 에이전트/도구 집합과 이름·역할이 다르며, 외부 스킬의 에이전트 정의를 직접 실행하지 못함 |
| `ooskill run`의 스킬 자체 최적화 | Copilot는 oo 스킬 문서를 읽을 수는 있어도, 스킬 내부의 자동 개선 규칙을 동일하게 실행하는 것은 별도 구현이 필요함 |
| `ooflow`의 전 단계 자동 오케스트레이션 | Copilot가 같은 단계명을 이해해도, OAIS의 순차 정책/중단 조건/커밋 규칙을 동일하게 강제하지 못함 |
| `oosync`의 환경 동기화 범위 | 현재 프로젝트 + 대상 경로라는 OAIS 고유 설정이 필요하고, Copilot 기본 환경만으로는 동기화 대상 판단이 불완전함 |
| `oostart`/`oostop`의 세션 생명주기 규칙 | OAIS의 세션 시작/종료 절차는 문서와 스크립트 기반이며, Copilot 일반 세션 규칙과 다름 |
| 프롬프트 1회 = premium request 1회 과금 해석 | Copilot CLI의 사용량 규칙이기 때문이며, OAIS 스킬 내부 카운트와는 별개 |

#### 왜 이런 차이가 생기나

이 차이는 단순 기능 부족이 아니라 구조 차이 때문이다.

1. **세션 상태 공유가 다르다.**  
   OAIS의 `oocontext` 같은 SP 상태는 Copilot 기본 세션에 자동 이식되지 않는다.

2. **에이전트/스킬 실행 주체가 다르다.**  
   OAIS 스킬은 “문서+스크립트+전용 에이전트” 조합으로 동작하지만, Copilot는 그 조합을 그대로 내부적으로 재현하지 않는다.

3. **파일명/문서 규칙이 프로젝트 전용이다.**  
   `d{SP}0004`, `d{SP}0010` 같은 규칙은 OAIS의 약속이라서, Copilot가 범용 규칙처럼 보장할 수 없다.

4. **자동화 정책이 다르다.**  
   `ooflow`처럼 중간 확인, 실패 에스컬레이션, 커밋 보존 같은 절차는 OAIS 문서가 정의한 정책이지 Copilot 기본 정책이 아니다.

5. **권한과 접근 범위가 다르다.**  
   Copilot는 현재 허용된 경로와 도구만 쓸 수 있고, OAIS의 다른 프로젝트/외부 시스템을 자동 포함하지 않는다.

### 2.5 이번 업데이트로 좋아진 점

예전에는 “전체 환경”, “권장”, “연동” 같은 말이 많아서 해석 여지가 컸는데,  
이제는 대부분의 oo 스킬이 스스로 다음을 말해준다.

- 무엇을 하는지
- 무엇을 하지 않는지
- 어디를 참조하는지
- 무엇을 바꾸는지
- 자동인지 반자동인지
- 어떤 에이전트와 맞는지

그래서 현재 기준으로는 **호환성보다 활용 가능 범위와 비활용 이유가 훨씬 선명해졌다**고 보는 게 맞다.

### 2.6 Claude 스킬 쪽에서 더 개선되면 좋은 점

업데이트 후에도 더 좋아질 수 있는 부분은 있다.

#### 1) 용어를 더 완전하게 통일

- 자동 / 반자동 / 수동
- 참조 범위
- 수정 대상
- 에이전트 호환

이 4개는 모든 oo 스킬에서 같은 순서와 같은 표현으로 보여주면 더 읽기 쉽다.

#### 2) 대체 경로를 더 명시

에이전트 이름이나 명령어가 환경에 따라 다를 수 있으므로, 스킬 문서에 다음 세 경로를 함께 표기하면 좋다.

예시 (`oofix` 기준):
```
기본:  Skill 도구 → oofix 호출
대체:  uv run python .claude/skills/oofix/scripts/oofix_run.py
수동:  d{SP}0004 이슈 목록 직접 확인 후 수정
```

이렇게 적어두면 Copilot가 **무엇을 못 쓰는지**뿐 아니라, **못 쓰는 이유 때문에 어떤 경로로 우회해야 하는지**까지 한 번에 이해할 수 있다.

#### 3) SP 예외 규칙을 더 짧게 요약

특히 `oofix`처럼 `d{SP}0004 + d0004`를 둘 다 다루는 예외는  
짧은 규칙 박스로 따로 넣으면 헷갈림이 줄어든다.

#### 4) 오케스트레이션과 실행을 더 분리

`ooflow`처럼 전체 흐름을 관리하는 스킬은  
“직접 구현하지 않음”을 더 강하게 써두는 편이 좋다.

### 2.7 결론

지금의 oo 스킬은 예전보다 훨씬 명확해졌고, Copilot CLI와 비교해도 **“무엇을 어디까지 다루는지”**가 잘 드러난다.  
다만 여전히 완전히 같은 환경은 아니므로, Copilot CLI에서는 **현재 프로젝트 기준 + 허용된 파일/도구 + 요청 단위 과금**으로 이해하는 것이 정확하고,  
활용이 안 되는 부분은 **SP 세션 공유 없음, 전용 에이전트 부재, 프로젝트 규칙 불일치, 자동화 정책 차이** 때문이라고 보면 된다.
