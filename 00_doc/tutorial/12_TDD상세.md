# 개발 시나리오: TDD 상세

> RED → GREEN → REFACTOR 사이클을 구체적으로 안내합니다.

---

## 개요

테스트 주도 개발(TDD)의 RED→GREEN→REFACTOR 사이클을 구체적으로 안내합니다. 먼저 실패하는 테스트를 작성하고, 최소 코드로 통과시킨 뒤, 리팩토링하는 반복 과정입니다.

**이럴 때 사용:** 새 함수/모듈을 견고하게 구현할 때, 테스트 커버리지를 높이고 싶을 때
**결과물:** 테스트 코드 + 검증된 구현 코드


## TDD 사이클 개요

```
RED (실패하는 테스트) → GREEN (최소 통과 코드) → REFACTOR (품질 개선)
                    ↑_________________________________↓ (반복)
```

---

## Phase 1: RED (테스트 작성)

```bash
ootest run         # 테스트 프레임워크 기반 테스트 생성
```

### 테스트 작성 원칙
- **하나의 테스트 = 하나의 동작** 검증
- 함수의 정상 케이스 + 엣지 케이스 + 에러 케이스
- 이 단계에서 구현 코드는 없으므로 **모든 테스트가 FAIL**

### 예시
```python
# tests/test_calculator.py
def test_add_positive():
    assert add(2, 3) == 5

def test_add_negative():
    assert add(-1, -2) == -3

def test_add_zero():
    assert add(0, 0) == 0
```

### 테스트 체크리스트
```bash
ootest checklist                    # 현재 체크리스트 확인
ootest add checklist "경계값 테스트"  # 항목 추가
```

---

## Phase 2: GREEN (최소 구현)

```bash
oodev run          # 구현 시작
```

### GREEN 원칙
- 테스트를 통과하는 **가장 단순한 코드** 작성
- 하드코딩도 괜찮음 (다음 테스트가 강제로 일반화시킴)
- 한 번에 하나의 테스트만 통과시키기

### 확인
```bash
ootest run         # 모든 테스트 PASS 확인
```

---

## Phase 3: REFACTOR (개선)

### 리팩토링 대상
- 중복 코드 제거 (DRY)
- 변수/함수 네이밍 개선
- 복잡한 로직 분리
- 불필요한 의존성 제거

### 리팩토링 안전망
```bash
ootest run         # 리팩토링 후 테스트 재실행 → 여전히 PASS인지 확인
oocheck run        # 코드 품질 분석
```

### 리팩토링 후에도 테스트 실패하면
- 리팩토링이 동작을 변경한 것 → 되돌리기
- 테스트 자체가 구현에 의존적이었던 것 → 테스트 수정

---

## 반복 사이클

```
새 기능 요구사항
  → RED: 새 테스트 추가 (실패)
  → GREEN: 구현 (통과)
  → REFACTOR: 개선
  → 다음 기능으로...
```

### 사이클당 권장 크기
- 테스트 1~3개 단위로 사이클 반복
- 한 사이클에 30분 이상 걸리면 기능을 더 잘게 분할
