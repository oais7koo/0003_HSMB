## 1. 왜 표준 디자인 시스템이 필요한가

SP04 표준 웹은 31개 페이지를 운영합니다. 페이지마다 색상·여백·카드 스타일을 따로 작성하면:

- 비주얼 일관성 깨짐 (개인도장 페이지와 카드매핑 페이지가 다른 디자인)
- 색상 변경 시 31개 페이지 모두 수정 필요
- Java 백엔드 이관 시 프런트 재사용 어려움
- 신규 개발자가 어떤 색·간격을 써야 할지 모름

**해결**: 토큰(색·여백) + 컴포넌트(카드·버튼) + 매크로(자주 쓰는 구조)를 한 곳에서 관리하고, 모든 페이지가 같은 자산을 참조하도록.

## 2. 디자인 시스템 자산 구조 한눈에

```
0002_CCone/
├── 00_doc/sp04/
│   └── d42500_상세설계_web_base_표준디자인시스템.md   ← 표준 SSOT 문서
│
├── 00_doc/tutorial/
│   ├── 40_디자인.md                          ← (이 문서) 프로세스 가이드
│   └── skills/oodesign.md                    ← oodesign 스킬 사용법
│
└── 04_api_server/web/
    ├── static/css/
    │   ├── main.css            ← 기존 (헤더·푸터·기본 레이아웃)
    │   ├── tokens.css          ← CSS 변수 SSOT (--primary, --border 등)
    │   └── components.css      ← 재사용 클래스 (.step-card, .stat-card 등 13종)
    └── templates/
        ├── base.html           ← 위 3개 CSS link 포함 (모든 페이지 자동 상속)
        └── _components.html    ← Jinja 매크로 (step_card, stats_grid)
```

| 자산 | 역할 | 누가 쓰는가 |
|------|------|------------|
| **tokens.css** | 색·여백·그림자 CSS 변수 정의 | 모든 페이지 자동 |
| **components.css** | 재사용 클래스 (.step-card 등) | 모든 페이지 자동 |
| **_components.html** | Jinja 매크로 (단계 카드·통계 그리드) | 신규 페이지 (선택) |
| **d42500 문서** | 토큰 카탈로그·체크리스트·DO/DON'T | 사람·AI 가이드 |
| **base.html** | 위 자산을 자동 로드 | extends 한 모든 페이지 |

## 3. 디자인 시스템 구축 프로세스 (1회성)

> 본 절은 새 프로젝트에 디자인 시스템을 처음 구축할 때 참조. SP04에선 이미 완료된 상태.

### Phase 1 — 토큰 추출

```
1. 참조 페이지 1~2개 선정 (예: card_slip_map)
2. 사용된 색·여백·그림자를 :root CSS 변수로 추출
3. web/static/css/tokens.css 파일에 정의
4. base.html에 <link rel="stylesheet" href="/web/static/css/tokens.css">
```

### Phase 2 — 컴포넌트 클래스 추출

```
1. 참조 페이지의 페이지 전용 vs 재사용 가능 스타일 구분
2. 재사용 가능 클래스(.step-card, .stat-card 등)를 components.css로 이동
3. base.html에 link 추가
4. 참조 페이지의 inline 스타일은 페이지 고유만 남기고 제거
```

### Phase 3 — 가이드 문서 작성

```
1. d42500_상세설계_web_base_표준디자인시스템.md 작성
2. 토큰 카탈로그 (각 변수의 용도·예시)
3. 컴포넌트 카탈로그 (HTML 스니펫 + 변형)
4. 신규 페이지 체크리스트 + DO/DON'T
5. 점진 마이그레이션 정책
```

### Phase 4 — Jinja 매크로 (선택)

```
1. _components.html에 자주 쓰는 구조 매크로 정의
   - step_card(num, title) — 단계 카드
   - stats_grid(...) — 결과 통계 3카드
2. 신규 페이지에서 {% from "_components.html" import step_card %} 사용
3. 단, premature abstraction 회피 — 패턴이 2~3개 페이지에서 안정화된 후 추출
```

### Phase 5 — oodesign 스킬 통합

```
1. .claude/skills/oodesign/SKILL.md description에 'd42500' 키워드 추가
2. SKILL.md "관련 문서" 영역에 d42500 SSOT 참조 + 명령 패턴 예시
3. d42500 §9에 oodesign 양방향 참조
4. 효과: "oodesign d42500 적용 — {페이지}" 명령이 자동 작동
```

## 6. 핵심 원칙 (DO / DON'T)

### ✅ DO

- 모든 색상은 `var(--*)` 토큰 사용
- 단계가 있는 페이지는 `.step-card` 사용
- 결과 통계는 `.stats-grid` + 3개 `.stat-card`
- 페이지 고유 스타일만 `extra_css` 블록
- 신규 페이지는 base.html extends

### ❌ DON'T

- inline 색상값 직접 작성 (`color: #265cdc` → `color: var(--primary)`)
- Bootstrap `card` 클래스와 `.step-card` 혼용 (한 페이지는 한 시스템만)
- main.css 변수 재정의 (tokens.css와 충돌)
- 랜딩/인트로 페이지에 d42500 강제 (독자 디자인 보유, 별건)
- premature 매크로 추출 (패턴 안정화 전 추상화 금지)

## 8. 트러블슈팅

### 문제: 클릭 시 색상이 안 바뀜
- **원인**: 페이지에 inline `color: #265cdc` 같은 직접 값 남아있음
- **해결**: `oodesign d42500 적용 — {페이지}` 재실행으로 var(--*)로 치환

### 문제: 메뉴 클릭 시 404
- **원인**: 메뉴 DB의 path와 라우트 URL 불일치 (R158-3 사례 참조)
- **해결**: 메뉴 DB `web_menu_master` 테이블 확인 + 코드 URL 정합화

### 문제: 회귀 테스트 실패
- **원인**: 클래스 치환 시 JS selector(`querySelector('.card')`)도 함께 영향
- **해결**: HTML 클래스 변경 시 JS selector도 확인. 회귀 실패 시 변경 범위 좁혀 재시도

### 문제: 시각이 깨짐
- **원인**: Bootstrap과 components.css 클래스 혼용
- **해결**: 한 페이지 내에서 같은 컴포넌트에 두 시스템 동시 적용 금지. 표준 클래스로 통일

## 9. 점진 마이그레이션 정책

기존 31개 페이지를 한꺼번에 마이그레이션하지 않습니다. **수정 시점에 함께 마이그레이션**:

| 케이스 | 마이그레이션 방식 |
|--------|------------------|
| 기능 수정/버그 픽스 | 같은 커밋에 디자인 마이그레이션 포함 |
| 신규 페이지 | 처음부터 표준 적용 |
| 디자인 리뉴얼 캠페인 | 도메인 단위(`auth`, `tax`, `work`) 일괄 처리 |
| 랜딩/인트로 | 강제 마이그레이션 ❌ (별건 처리) |

이 정책의 이유:
- 회귀 위험 최소화 (큰 변경 한 번보다 작은 변경 여러 번이 안전)
- 비즈니스 우선 영역(auth, tax)부터 자연스럽게 표준화
- 리소스 분산 가능
