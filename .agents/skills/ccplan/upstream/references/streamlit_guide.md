## 1. Streamlit 프로젝트 계획 특화

### 1.1 3계층 문서 체계

Streamlit 프로젝트는 단순 Plan이 아닌 **페이지별 통합 개발서(단위개발 문서)**를 사용한다:

```
PRD (What: 무엇을)
  ↓
Plan (Where: 어디에 배치할 것인가)
  ↓
개발서 d{SP}1XXX_페이지명_단위개발.md (How: 어떻게 구현할 것인가)
```

| 문서 | 역할 | 수준 |
|------|------|------|
| `d{SP}0001_prd.md` | 요구사항 정의 | 전체 프로젝트 |
| `d{SP}0002_plan.md` | 구조/WBS/일정 | 페이지 구조 |
| `d{SP}1XXX_*_단위개발.md` | 페이지별 상세 설계 | 단일 페이지 |

### 1.2 개발서 파일 네이밍

```
d{서브프로젝트번호}{4자리순번}_{페이지명}_단위개발.md
```

**예시**:
- `d21720_앱_시니어복지_단위개발.md` (SP02, 1720번)
- `d21721_앱_복지카드_단위개발.md`
- `d22410_세무문의요청_단위개발.md` (SP02, 2410번)

### 1.3 통합 개발서 구조 (7섹션)

```
d{번호}_{페이지명}_단위개발.md
├── 1. 개요 (목적, 사용자, 주요 기능)
├── 2. PRD (화면 구성, 기능 목록, 데이터 요구사항)
├── 3. 플랜 (기술적 접근법, WBS, 태스크, 실행 순서)
├── 4. 설계 (레이아웃, DB 스키마, 로직 흐름, oo 모듈)
├── 5. 구현 (완료 후 작성 - 코드 스니펫, 특이사항)
├── 6. 검증 (테스트 결과, 체크리스트)
└── 7. 관련 문서
```

## 2. Streamlit Plan 수립 (Phase 3)

### 2.1 계획 수립 워크플로우

```
PRD 분석 → 페이지별 개발서 생성 → Task 매핑 → WBS 작성 → 스프린트 계획
```

### 2.2 WBS 형식 (Streamlit 특화)

```markdown
### E001: [페이지 그룹명]
  - F001-1: [페이지명] 페이지
    - F001-1.1: 사이드바 필터/검색 구현
    - F001-1.2: 메인 데이터 테이블 구현
    - F001-1.3: CRUD 기능 구현
    - F001-1.4: 데이터 연동 완료
    - **개발서**: d{SP}1XXX_{페이지명}_단위개발.md
```

### 2.3 플랜 출력물 형식

개발서 섹션 3에 작성:

```markdown
### 3.1 기술적 접근법
- 프레임워크: Streamlit + [DB 라이브러리]
- oo 모듈 활용: oo.ui, oo.db, oo.data_processing
- 상태 관리: st.session_state

### 3.2 WBS
| Phase | Task | 설명 | 우선순위 | 예상 시간 |
|-------|------|------|---------|---------|
| 1. 사이드바 | 필터 구현 | 검색/상태 필터 | High | 1h |
| 2. 메인 | 테이블 구현 | 데이터 표시 | High | 2h |
| 3. CRUD | 등록/수정/삭제 | 폼 + DB 연동 | High | 3h |

### 3.3 실행 순서
1. 데이터 조회 로직 (DB 연동)
2. 사이드바 필터 구현
3. 메인 테이블 표시
4. CRUD 기능 추가
5. 세션 상태 연동
```

## 3. UI/UX 설계 (Phase 4)

### 3.1 설계 워크플로우

```
플랜 확인 → 사용자 워크플로우 파악 → 정보 구조 설계(IA)
→ 와이어프레임 → DB 스키마 → Streamlit 컴포넌트 매핑
→ 상호작용 흐름 정의 → 설계 문서 작성
```

### 3.2 레이아웃 패턴

#### CRUD 페이지 패턴

```
+----------------------------------------+
| 헤더: 페이지 제목                       |
+----------+-----------------------------+
| 사이드바 | 메인 영역                    |
| - 검색   | 상단: 액션 버튼              |
| - 필터   | 중단: 데이터 테이블          |
|          | 하단: 상세/폼               |
+----------+-----------------------------+
```

#### 대시보드 패턴

```
+----------------------------------------+
| 헤더: 대시보드                          |
+----------+-----------------------------+
| 사이드바 | KPI 1 | KPI 2 | KPI 3       |
| - 기간   | 차트 영역                    |
| - 필터   | 상세 테이블                  |
+----------+-----------------------------+
```

#### 탭 분리 패턴

```
+----------------------------------------+
| 헤더: 페이지 제목                       |
+----------+-----------------------------+
| 사이드바 | [탭1] [탭2] [탭3]           |
| - 공통   | 탭별 콘텐츠 영역             |
| - 필터   |                             |
+----------+-----------------------------+
```

#### 조회/분석 패턴

```
+----------------------------------------+
| 헤더: 분석 제목                         |
+----------+-----------------------------+
| 사이드바 | 상단: 요약 메트릭            |
| - 기간   | 중단: 차트/그래프            |
| - 카테고리| 하단: 상세 데이터 테이블    |
+----------+-----------------------------+
```

### 3.3 컴포넌트 매핑 테이블

| 기능 | Streamlit 컴포넌트 | 배치 위치 | 비고 |
|------|-------------------|---------|------|
| 텍스트 검색 | `st.text_input()` | 사이드바 | placeholder 활용 |
| 상태 필터 | `st.selectbox()` | 사이드바 | 전체 옵션 포함 |
| 날짜 범위 | `st.date_input()` | 사이드바 | 시작/종료 |
| 다중 선택 | `st.multiselect()` | 사이드바 | 카테고리 필터 |
| 토글 스위치 | `st.toggle()` | 사이드바 | on/off 필터 |
| 데이터 표시 | `st.dataframe()` | 메인 | 읽기 전용 |
| 데이터 편집 | `st.data_editor()` | 메인 | 인라인 편집 |
| 상세 정보 | `st.expander()` | 메인 | 접기/펼치기 |
| 탭 분리 | `st.tabs()` | 메인 | 기능 그룹화 |
| 폼 입력 | `st.form()` | 메인/모달 | 일괄 제출 |
| KPI 카드 | `st.metric()` | 메인 상단 | 요약 지표 |
| 차트 | `st.bar_chart()`, `st.line_chart()` | 메인 | 기본 차트 |
| 알림 메시지 | `st.success()`, `st.error()` | 메인 | 작업 결과 |
| 컬럼 분리 | `st.columns()` | 메인 | 나란히 배치 |

### 3.4 DB 설계 표준용어 적용

DB 컬럼/테이블 설계 시 `.claude/skills/oodb/templates/db_standard_word.md` 준수.

**주요 표준 약어**:

| 카테고리 | 약어 | 예시 |
|---------|------|------|
| 식별자 | id, seq, cd, no, key | user_id, item_seq |
| 명칭 | nm, ttl, lbl, desc, cont | user_nm, page_ttl |
| 일시 | dt, tm, dttm, yr, mon | reg_dt, upd_dttm |
| 상태 | stat, typ, cat, lvl, flg | use_flg, proc_stat |
| 수량 | cnt, amt, qty, rate | view_cnt, pay_amt |

**자주 사용하는 컬럼명 패턴**: `*_id`, `*_cd`, `*_nm`, `*_dt`, `*_yn`, `*_no`, `*_cnt`, `*_amt`

### 3.5 설계 문서 형식 (개발서 섹션 4)

```markdown
### 4.1 레이아웃 구조
[ASCII 와이어프레임 - CRUD/대시보드/탭 패턴 중 선택]

### 4.2 컴포넌트 상세
| 컴포넌트 | 위치 | Streamlit 코드 | 데이터 소스 |
|---------|------|--------------|------------|

### 4.3 DB 스키마
| 테이블명 | 컬럼 | 타입 | 설명 |
|---------|------|------|------|

### 4.4 데이터 흐름
[DB 조회 → 필터 적용 → 화면 렌더링 흐름 다이어그램]

### 4.5 oo 모듈 활용 계획
| 모듈 | 함수 | 용도 |
|------|------|------|
```

## 4. 서브에이전트 활용

### 4.1 플랜 에이전트 호출 (Phase 3)

```
Task(
  subagent_type="streamlit-page-planner",
  prompt="다음 페이지의 구현 계획을 수립하라:
  작성 항목:
  - 기술적 접근법 (Streamlit + DB)
  - WBS (Phase/Task 단위)
  - 상세 태스크 목록 (예상 시간 포함)
  - 리스크 관리 (외부 API, DB 연동 등)
  - 실행 순서 (의존성 기반)"
)
```

### 4.2 설계 에이전트 호출 (Phase 4)

```
Task(
  subagent_type="streamlit-page-designer",
  prompt="다음 페이지의 상세 설계를 수행하라:
  작성 항목:
  - 레이아웃 구조 (ASCII 와이어프레임)
  - 사이드바/메인 컴포넌트 목록
  - DB 스키마 (표준용어 적용)
  - 데이터 흐름 다이어그램
  - oo 모듈 활용 계획
  - 상호작용 흐름 정의"
)
```
